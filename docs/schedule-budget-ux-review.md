## 자동 fix (이 PR)

### Fix 1. 스케줄 D-Day 카드 더미 날짜 → 행동 유도 문구
`Schedule.tsx`. 결혼일 미설정·미정도 아닌 신규 사용자에게 노출되던
`0000년 0월 0일 (0요일)` 더미 문자열을 제거하고, 카드 전체가 이미
`/my-schedule`로 이동하므로 `결혼식 날짜를 설정해보세요 →`(primary 색)
로 교체.

### Fix 2. 스케줄 정리 팁 빈 아이콘 자리 제거
`Schedule.tsx`. 이모지 제거 후 남은 빈 `<span aria-hidden>`을 삭제해
`flex gap` 으로 생기던 어색한 여백 정리.

### Fix 3. 일정 추가 시 단계 기반 날짜 자동 제안
`MySchedule.tsx`. 타임라인 단계(phase-1~5)를 선택하면, 날짜가 비어 있고
결혼일이 설정돼 있을 때 `결혼일 − 단계 기본 리드타임`
(`TIMELINE_PHASES[].defaultDaysBeforeWedding`)으로 날짜를 자동 입력.
과거 날짜가 나오면 오늘로 보정. 사용자가 직접 덮어쓸 수 있음.

### Fix 4. `getTaskUrgency` 헬퍼 구현 + 지남 뱃지
`lib/schedule.ts`, `Schedule.tsx`. `schedule.test.ts`에 명세만 있고
구현이 없어 **5개 테스트가 red 상태**였던 `getTaskUrgency`를 구현
(past_due / urgent(≤7) / this_month(≤30) / later). "다가오는 일정"에서
지난 일정에 빨간 `지남` 뱃지를 붙여 임박/연체를 한눈에 구분(예산 화면의
`X일 지남` 패턴과 통일).

### Fix 5. 예산 화면 이모지 제거 잔재 정리
`Budget.tsx`. ① 단계별 가이드의 빈 아이콘 `<span>`을 `Sparkle` 아이콘으로
교체, ② 양가 분담 라벨(`공동`/`신랑측`/`신부측`)과 카톡 공유 텍스트
첫 줄의 앞쪽 공백 제거.

## 결정 필요 (사용자 보고)

### A. "다가오는 일정"의 지난 일정 처리 범위
**현 상태**: `upcomingTasks`는 미완료 항목을 날짜 오름차순으로 잘라 상위 4개
표시. 추천 템플릿으로 시드된 오래된 미완료 항목이 영원히 상위 4슬롯을
차지해, 막바지 사용자(박지훈 페르소나)에게 실제 임박 일정이 안 보일 수 있음.
(이번 PR 은 "지남" 뱃지로 가시성만 개선)

**선택지**:
- **A1** — 그대로 (가장 시급=가장 오래된 연체부터 노출, 정렬상 자연스러움)
- **A2** — "지난 일정" 별도 섹션으로 분리, 다가오는 일정은 미래만
- **A3** — N일(예: 90일) 이상 지난 항목은 상위 목록에서 숨기고 카운트만 표시

→ **추천 A1 유지 + 이번 PR 의 지남 뱃지로 충분**. 다만 연체 항목이 많은
사용자 데이터가 쌓이면 A2 재검토.

### B. 예산: "다가오는 결제 예정" vs "다가오는 잔금 일정" 카드 중복감
**현 상태**: 일정에서 파생된 `다가오는 결제 예정`(미완료 일정 → 예산 기록
유도)과 지출의 잔금에서 파생된 `다가오는 잔금 일정`(잔금 결제) 두 카드가
인접 노출. 둘 다 D-day 뱃지 + 액션 버튼이라 비슷해 보일 수 있음.

**선택지**:
- **B1** — 그대로 (출처·액션이 달라 분리 보존 가치 있음)
- **B2** — "다가오는 결제" 한 섹션으로 통합, 잔금/일정 출처를 작은 라벨로 구분

→ **추천 B1**. 액션(예산 기록 vs 잔금 결제 완료)이 본질적으로 달라 분리가 자연스러움.

### C. `WeddingInfoSetupModal` 공유 (트랙 2)
**현 상태**: 일정/예산 모두 이 모달을 onboarding/스타일 편집에 사용. 트랙 2 와
공유될 수 있어 **이번 PR 에서는 모달 자체를 건드리지 않고** 페이지 레벨에서만
개선함. 모달 수정이 필요한 항목은 발견되지 않음.

→ 모달 수정 없음. 트랙 2 머지 후 rebase 필요시 충돌 없음.

## 그 외 발견 (작은 것, 보고만)

- **`getDateUrgency`(색상)와 `getTaskUrgency`(의미 분류)의 임계값이 다름**
  (전자 ≤3 주황/≤7 노랑, 후자 ≤7 urgent). 색상 단계는 더 세밀한 게
  의도된 것으로 보여 통일하지 않음. 필요시 후속 정리.
- **`placeValueTags.test.ts` 1개 실패**는 places 도메인(검토 범위 밖)으로
  base 에서도 이미 red. 이 PR 과 무관.
- **`Budget.tsx`의 `as any` 2곳**(`handleMarkBalancePaid`, `BudgetAddSheet`
  onSave)은 기존 lint 에러. 이번 변경과 무관해 보존.

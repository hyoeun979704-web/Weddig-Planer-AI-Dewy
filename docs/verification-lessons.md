# 검증 회귀 사례 (상세)

> CLAUDE.md "검증" 섹션의 회귀 스토리 전문. 같은 실수를 반복하지 않기 위한 기록.

## 회귀 — label vs value 혼동 (Round 13)

- 사용자 요청: "전라남도 → 전남, 충청북도 → 충북 통일"
- 잘못된 적용: UI label 과 backend value 둘 다 약자로 통일
- 결과: `ILIKE '%충남%'` 가 `"충청남도"` 매칭 안 됨 (비연속 글자) → 충남/충북/전남/경남/경북 필터 전부 0건
- 검증 실패: SQL row count 만 확인하고 정상 보고
- 진짜 검증: 사용자가 실제 앱 스크린샷으로 0건 표시 보여줌

**규칙**: label (사용자 표시) vs value (백엔드 매칭) 의 의미를 항상 분리해서 생각.
사용자 표시 문구 변경 요청 ≠ 백엔드 검색 키워드 변경.

## 회귀 — DB 스키마 정합성 미확인 (Round 14)

- 사용자 보고: "충남 필터 0건"
- 추정 원인 (잘못): 데이터 sparsity 라고 판단
- 진짜 원인: `useWeddingSchedule` SELECT 가 DB 에 없는 컬럼 15개(role, country,
  persona_mode, wedding_venue_* 등) 요청 → PostgREST 422 에러 → 전체 쿼리 실패 →
  weddingSettings empty → 필터 흐름 차단
- 검증 실패: 코드와 마이그레이션 파일만 보고 "DB 에 당연히 있다" 가정. DB 실제 스키마
  미확인. 사용자가 e2e 결과 보여줄 때까지 못 잡음
- 진짜 검증: `SELECT column_name FROM information_schema.columns WHERE table_name=...`
  으로 실제 컬럼 list 확인

**규칙**: DB 의존 코드 검토 시 **항상 `list_tables` / `information_schema` 로 DB 실제
스키마 먼저 확인.** 마이그레이션 파일 존재만으로 안심 금지. `supabase_migrations.
schema_migrations` 의 적용 history 와 repo 의 마이그레이션 파일 수가 다르면 즉시
경고 + 적용 가능한 것 식별.

## 회귀 — import 를 동명 const 로 섀도잉 → 런타임 ReferenceError (코드리뷰 라운드)

- 작업: 공통화 리팩터(`_shared/supabase.ts` 의 `adminClient()` 로 15개 함수 통합)
- 잘못된 적용: `import { adminClient } from "..."` 해놓고 같은 이름으로
  `const adminClient = adminClient();` — 블록스코프 const 가 import 를 섀도잉,
  우변 호출이 **TDZ(temporal dead zone)** 에 걸림
- 결과: kakao-pay-approve / kakao-pay-charge-approve 가 매 호출 즉시
  `ReferenceError: Cannot access 'adminClient' before initialization` → **구독·하트충전
  결제 승인 100% 불능**. 인증 통과 직후 결제 쓰기 직전에 throw
- 검증 실패: `npm run build`·`npm run lint`·esbuild 전부 **통과**. 유효한 문법·합법적
  JS 라 정적 검사로는 안 잡힘. 런타임 호출/e2e 결제만이 잡을 수 있었음
- 진짜 검증: 동명 섀도잉을 최소 Node repro 로 재현 → 다른 13개 함수는 로컬명이
  `admin`/`supabaseAdmin` 으로 달라 무사함을 확인. 로컬 변수를 `admin` 으로 rename

**규칙**: import 심볼을 **같은 이름의 지역 변수로 절대 재선언 금지**(섀도잉=TDZ 폭탄).
빌드·린트·esbuild 통과는 **타입/문법 검증일 뿐 런타임 안전 보증이 아니다** — 특히
결제·인증처럼 호출 경로를 직접 안 밟는 코드는 "정적 통과"만으로 완료 보고 금지.
멀티라인 메서드 체인(`client\n  .from(...)`)은 단순 텍스트 치환으로 놓치기 쉬우니
rename 후 `grep` 으로 남은 참조 0건 확인.

## 회귀 — 원격에서 추측으로 "수정" 반복 + 사용자 검증 의심 (꽃 머지 게임 흰 화면)

- 작업: 게임에 무료3+광고3 일일 쿼터 UX 추가(#226). 사용자가 인앱 웹뷰(카톡)에서
  게임 영역이 **흰 화면**(쿼터만 차감, 캔버스 안 뜸)이라고 스크린샷으로 3회 보고.
- 잘못된 대응(반복):
  1) e2e(브라우저) 검증이 **불가능한 샌드박스**인데도, 정적 통과(tsc·build·lint)만
     보고 "고쳤다"는 톤으로 가설 기반 커밋을 연달아 푸시(forwardRef 제거 → 캔버스
     사이징 교체). 각 커밋이 **실제로 렌더되는지 확인 못 한 채** 사용자에게 떠넘김.
  2) 사용자가 "여전히 안 나와"라고 하자 **"캐시된 옛 빌드를 본 것"이라고 사용자의
     검증을 의심**. 사용자는 이미 캐시를 다 확인하고 보고한 상태였음.
- 왜 나쁜가: 인터랙티브 캔버스 게임은 이 환경에서 직접 렌더 확인이 **원천적으로 불가**.
  "작동한다 ≠ 검증됨" 규칙 위반(정적 통과를 렌더 검증으로 착각). 게다가 사용자의
  실기기 확인을 의심하는 건 신뢰를 깨고 디버깅을 더 늦춤.

**규칙(2가지)**:
1. **사용자의 실환경 확인을 의심하지 말 것.** "안 된다면 안 되는 것." 캐시/사용자
   실수 탓으로 돌리는 추측을 **입에 올리지 말 것**(사용자가 이미 확인했다). 의심 대신
   내 코드/배포 경로에서 객관적 근거(빌드 로그·배포 sha·CI 상태)를 먼저 확인.
2. **렌더 검증 불가 영역(캔버스 게임 등)에서 가설 커밋을 연달아 푸시 금지.** 한 번에
   하나씩, **사용자가 검증할 수 있는 형태**로 바꾸고 결과를 받기 전엔 "고쳤다" 톤 금지.
   막히면 추측 누적 대신 **검증된 베이스라인으로 되돌려** 변수부터 0으로 만든다.

## 회귀 — PL/pgSQL RETURNS TABLE 변수가 컬럼을 가리는 ambiguous 에러 (260611)

- 사용자 보고: "공유 기능이 애초에 안 됨" (청첩장 발행 전멸 — 모든 행 draft·slug NULL)
- 진짜 원인: `publish_invitation` 의 `RETURNS TABLE(... share_slug ...)` OUT 변수가
  본문 `EXISTS (SELECT 1 FROM invitations WHERE share_slug = v_new_slug)` 의 비한정
  컬럼과 충돌 → 42702 ambiguous. **CREATE FUNCTION 은 통과**하고, 기존 slug 재사용
  경로는 멀쩡해서 첫 발행(신규 slug 생성) 경로만 100% 런타임 실패.
- 검증 실패: RPC "존재 확인"만 하고 정상 보고. 함수 존재 ≠ 함수 실행 가능.
- 진짜 검증: `SET LOCAL ROLE authenticated; SET LOCAL request.jwt.claims='{"sub":...}'`
  롤백 트랜잭션으로 실제 호출 경로를 실행해 슬러그 발급까지 확인.

**규칙**: ① RETURNS TABLE / OUT 파라미터 이름과 같은 컬럼을 본문에서 쓸 때는 항상
테이블 별칭으로 한정. ② DB 함수는 정의 조회가 아니라 **실 역할(role+jwt claims)로
호출**해 봐야 검증된 것. 분기(신규/재사용)별로 한 번씩.

## 회귀 — 동작하는 척하는 placeholder CTA (죽은 토스트) (260614)

- 사용자 보고: "이전 감사 때 활성화 안 되는 토스트 확인하라고 했는데 아직도 남아있네."
  상세페이지 '예약 문의' 버튼이 미입점 업체에선 `toast.info("아직 앱 문의를 받지 않는
  업체예요")` 만 띄우고 실제 연결을 안 함. 미입점이 활성 업체의 **99% 이상**(4,126곳 중
  소유자 있는 곳 극소수)이라, 사실상 거의 모든 상세페이지에서 '문의'가 죽어 있었음.
- 왜 **전체 감사가 놓쳤나**: 감사 6차원이 보안·P0버그·성능 중심이라, 이 버튼은
  **에러 없이 토스트를 띄우니 "정상 동작"으로 통과**. 하지만 사용자가 기대하는 동작(업체에
  문의 연결)을 전혀 안 함 = placeholder/미완. "기능 존재(클릭하면 뭔가 뜸) ≠ 기능 완성".
  코드만 보면 멀쩡(빈 catch·await 누락·SQL 인젝션 없음)해서 정적·보안 렌즈로는 안 걸린다.
- 진짜 원인은 사실 **데이터/제품 결정 부재**였다: 미입점 업체를 어디로 연결할지(전화·SNS)
  정해두지 않아 "안내 토스트"로 때워둔 채 방치. 연락 필드(`place_details.tel`·`kakao_channel_url`
  ·`naver_place_url`·`website_url`·`instagram_url`)는 **이미 수집돼 있었는데** 버튼이 안 썼다.
- 수정: 미입점은 온라인 채널→전화→안내 폴백, 입점은 사장님이 채널 선택(chat/url/phone).

**규칙**: ① **주요 CTA(문의·예약·구매·공유)는 `toast`/안내로 끝나면 placeholder 로 간주**,
완료 보고 금지하고 별도 추적. 누를 때 기대 동작을 끝까지 수행하는지 페르소나 walkthrough 로
확인. ② **전체 감사·코드리뷰에 'dead-end UI' 차원을 명시적으로 추가**(보안·버그만 보지 말 것):
no-op onClick, "준비 중"/"아직 ~안 함" 영구 안내, 비활성처럼 보이는 버튼을 grep(`toast.info`
+CTA 라벨)으로 훑는다. ③ placeholder 의 뿌리가 **제품/데이터 결정 부재**면, 코드 패치 전에
"이 버튼은 무엇을 해야 하는가"부터 정한다(이미 가진 데이터로 되는 경우가 많다).

## 회귀 — RPC 호출 인자 ≠ 함수 시그니처 → 100% 실패 (업체 승인 불능, 260615)

- 사용자 보고: "업체정보 승인이 여전히 안 됨"(전체 감사 후에도 잔존).
- 진짜 원인: 클라(`AdminBusinessReview.reviewListing`)가 `admin_review_listing` 을 **3인자**
  (`p_place_id, p_approved, p_note`)로 호출하는데 DB 함수는 **2인자**뿐 → PostgREST 가
  named-arg 로 함수를 못 찾아(PGRST202) 호출이 **100% 실패** → "처리에 실패했어요"로 승인 불능.
- 왜 감사가 놓쳤나: **빌드·린트·타입체크 전부 통과**한다. `(supabase as any).rpc(...)` 캐스트가
  타입 검출을 가리고, 정적 분석은 클라 인자와 DB 함수 시그니처의 **교차 정합성**을 안 본다.
  관리자 승인 같은 **호출 경로를 직접 안 밟으면** 런타임 전까지 드러나지 않는다.
  (연관: 결제 섀도잉 TDZ, publish_invitation ambiguous — 모두 "정적 통과 ≠ 런타임 안전".)
- 곁가지 정합성: 사장님 페이지는 반려 사유로 `places.moderation_note` 를 읽는데 컬럼이
  아예 없었다(항상 빈값). 코드가 참조하는 컬럼의 DB 존재 미확인(스키마 정합성 규칙 위반).

**규칙**: ① **RPC 시그니처를 바꾸거나 클라 호출을 짤 때, 모든 `.rpc("name", {...})` 호출부의
인자 집합이 함수 파라미터와 정확히 일치하는지 교차 확인**(PostgREST 는 named-arg 매칭 — 빠진/
남는 인자 하나면 함수 미발견으로 전량 실패). `as any` 캐스트는 이 불일치를 숨기므로 더 위험.
② **DB 함수는 정의 조회가 아니라 실 역할(role+jwt claims)로 호출**해 봐야 검증된 것(이번엔
admin 역할 시뮬레이션으로 승인 반영까지 확인). ③ 검토 RPC 군은 **시그니처를 통일**(전부
`p_*, p_approved, p_note`)해 호출부 드리프트를 줄인다.

## 회귀 — 신선도 신호 오류: updated_at ≠ last_source_date (260615)

- 사용자 요청: "데이터 신선도 백엔드 정합성 확인".
- 발견: 어드민 신선도 대시보드가 `updated_at`(정규화 트리거·모더레이션·**마이그레이션** 등
  무엇이든 손대면 갱신)으로 신선도를 계산 → **모든 카테고리가 stale 0 = '신선'으로 거짓 표시**.
  실제 수집일 `last_source_date` 기준으론 studio 44·hanbok 37·wedding_hall 17곳 등이 60일+.
  게다가 `last_source_date` 자체가 대부분 NULL(수집 파이프라인이 소스 발행일만 넣고, 없으면 미기록).
- 왜 위험: 쿼리도 통과하고 숫자도 나오니 "정상"으로 보인다. 하지만 **무엇으로 재느냐(신호 선택)**
  가 틀리면 지표가 항상 낙관적으로 거짓말한다(=label vs value 의 시간축 버전).

**규칙**: ① **"최신성/신선도"는 '마지막으로 손댄 시각'(updated_at)이 아니라 '데이터를 마지막으로
수집·검증한 시각'으로 재야 한다.** 트리거·배치·마이그레이션이 updated_at 을 흔들면 신호가 오염된다.
② 신호 컬럼이 NULL 다발이면 "알 수 없음"으로 **정직하게 분리 표기**(거짓 '신선' 금지). ③ 근본은
파이프라인이 수집 시각을 기록하게 하는 것 — 단, 의미 변경은 그 컬럼을 쓰는 곳(scoring·dedupe)까지
영향분석 후 결정.

# 업체(벤더) 상세·혜택·커머스 통합 기획 (260619)

> 목적: 업체 상세페이지 + 상품/이벤트/쿠폰 등록·노출 + 문의/결제 + 비용 안내까지를 한 번에
> 설계하고 **단계별로** 진행하기 위한 단일 기획 문서. (세부 카테고리 구조 감사는
> `docs/260619_vendor_detail_audit.md` 참조.) 코드 기준 실태 반영.

## 0. 목표
- **소비자**: 업체 상세에서 정보·혜택(쿠폰/이벤트)을 보고 → 문의 또는 구매로 전환.
- **업체**: 상품/이벤트/쿠폰을 쉽게 등록(이미지 기반 포함) → 노출·결제·문의로 매출.
- **플랫폼(대표)**: 초기 제휴 무료 정책 하에서 **마이너스 없이** 운영. 결제수수료는 PG 패스스루.

## 1. 현재 상태 (코드 기준 — 되는 것/안 되는 것)
- 상세 탭: 기본정보 / 상세정보 / 리뷰 (`PlaceDetailLayout`).
- 혜택: 쿠폰·이벤트는 **기본정보 탭 혜택군**에 노출(이번 세션 반영). 상품·포트폴리오는 상세정보 탭(`PlaceBusinessSections`).
- 카테고리별 상세(`CategoryExtras`) 풍부도 격차 큼, 기타(etc)=폴백 추가. 기본정보는 카테고리 공통 shell(핵심 스펙 없음). → `docs/260619_vendor_detail_audit.md`.
- **결제 시스템**: 카카오페이 연동 존재(`kakao-pay-order-ready/approve` 등). 단 **쇼핑 `products` 테이블 전용** — 업체상품 `business_products`는 **미연결(구매 불가)**.
- `business_products` 컬럼: id·name·price·description·image_url(1장)·moderation_status. **결제지원 플래그·다중 이미지 없음.**
- 인앱 문의: `PlaceInquirySheet`(place_inquiries 생성) 존재 → 재사용 가능.
- 이벤트: 등록(`BusinessEvents`, 이미지 기반 가능) → 홈 '이벤트' 탭(`/deals` 전체 혜택)에 노출(이번 세션). **노출 위치 사용자 최종 확인 필요.**

## 3. 타깃 설계
### 3.1 업체 상세페이지 정보구조
- **기본정보 탭**: 갤러리 → 이름·평점·태그 → **카테고리별 핵심 스펙 요약(신규)** → 혜택(쿠폰·이벤트) → 업체 소개 → 연락/위치 → CTA.
- 핵심 스펙: 카테고리→`[{label,value}]` 맵(가격/식대/원본컷/캐럿 등). 값 없으면 생략. (표=감사문서)
- 상세정보 탭: 포트폴리오·상품(상세 링크)·카테고리 상세.
### 3.2 상품 (`business_products`)
- 상세페이지 `/product/:id`: 썸네일(없으면 업체 대표이미지)·다중 상세 이미지·설명·관련 포트폴리오.
- 등록 폼: 썸네일 + **상세 이미지 다중** + 설명 + **결제지원 토글** + 가격.
- CTA(하단):
  - 결제지원 ON → **[구매하기]**(카카오페이) + **[문의하기]**
  - 결제지원 OFF → **[문의하기]**
  - 문의하기 → **전화(tel:)** / **인앱 채팅(PlaceInquirySheet)** 선택.
### 3.3 이벤트
- 등록(이미지 기반)·노출(홈 이벤트탭, 정렬 상세>텍스트·제휴우선)·상세페이지(`/event/:id`)·썸네일 폴백 = #362.
- 상세페이지 CTA: [업체 확인하기] / [신청하기] — **신청하기 동작 확정 필요**(문의? 결제? 외부?).
### 3.4 쿠폰: 현행 + 기본탭 노출(완료). (다운로드/사용 플로우는 현행 유지.)
### 3.5 문의: 전화 + 인앱 채팅(PlaceInquirySheet) 공통 컴포넌트화(상품/이벤트/업체 공용).

## 4. 결제 연동 설계 (카카오페이)
- 스키마: `business_products` + `payment_enabled boolean default false`, `detail_images text[]`.
- edge function `kakao-pay-order-ready`(+approve): `products`에 없으면 **`business_products`** 조회(approved + payment_enabled)해 가격·이름으로 주문 생성. order_items.product_id에 업체상품 id 저장(스냅샷).
- 플로우: 상세 [구매하기] → ready(서버 금액 산정) → 카카오 결제 → approve → 완료/주문내역.
- ⚠️ **제약**: 본 sandbox는 Supabase·카카오 PG 차단 → **실결제 e2e 불가**. 코드·esbuild 정적검증까지만, 실결제는 **실환경(배포) 검증 필요**. (결제는 되돌리기 어려움 — 실환경 테스트 필수.)

## 5. 비용 안내 가이드 설계
- 모델: **업체 수령액 = 판매가 − 결제수수료 − (결제수수료의 부가세 10%) − 플랫폼 수수료.**
  - 초기 제휴 업체: **플랫폼 수수료 = 0** → 결제수수료(+VAT)만 차감 = "최저치" 안내.
  - 플랫폼은 결제수수료를 떼지 않고(PG가 가져감) 0을 받으므로 **마이너스 없음**.
- 표기: 상품/이벤트/쿠폰 등록 폼에 안내 컴포넌트. 예) "10만원 상품 → 결제수수료 약 1,620원(+VAT 162원) 차감, 수령 약 98,218원. 듀이 입점 수수료: 초기 제휴 무료."
- **카카오페이 수수료율(조사 반영, §9)**: 일반 가맹점 카드 **1.62%**, 카카오페이머니(선불) **1.45%**, 영세 가맹점 **0.38%**(우대구간 차등). → **안전 안내치 = 일반 1.62% + VAT 10%**(보수적, 실제는 더 낮을 수 있어 업체가 손해 안 봄). 코드엔 **설정 상수**로 두고 계약 확정 시 조정.

## 6. 단계별 진행 계획 (각 단계 = 별도 PR, 순차)
- **Phase A — 상품 상세페이지 + 문의하기** (결제 무관, 검증 가능): `/product/:id`, 썸네일/설명/상세이미지/관련 포트폴리오, [문의하기](전화/인앱채팅). 상품 카드→상세 이동.
- **Phase B — 기본정보 구조화**: 카테고리별 핵심 스펙 요약 블록 + 업체 소개 노출. (감사문서 Phase 1)
- **Phase C — 카카오페이 구매**: 스키마(payment_enabled/detail_images) + edge function 확장 + [구매하기] + 결제 폼 안내. (실환경 검증 필요)
- **Phase D — 비용 안내 가이드**: 수수료율 확정 후 등록 폼 안내 컴포넌트.
- **Phase E — 마감**: 얇은 카테고리 상세 보강, 이벤트 위치/신청하기 최종 반영, 회귀 점검.

## 7. 진행 전 확정 필요 (사용자)
1. **이벤트 노출 위치** 최종(홈 '이벤트' 탭 = `/deals` 전체 혜택이 맞는지).
2. **이벤트 '신청하기'** 동작(문의/결제/외부 링크 중).
3. **결제지원 판별** = 상품별 토글로 OK?
4. **카카오페이 결제수수료율(%)**.
5. **카테고리별 핵심 스펙** 확정(감사문서 표 그대로/가감).
6. **#362 처리**: 지금까지(상품 상세 모달·라이트박스·혜택 노출·이벤트 etc) 머지할지, 본 기획에 맞춰 일부 조정할지.

### 9.1 조사 요약
- **상세페이지 정보구조**: 아이웨딩은 화보·**추가비용**·주소·**발렛(주차)비**·영업시간 등 "실질 정보"를 노출하고, 스드메를 **장바구니+쿠폰**으로 담아 **투명가격** 비교를 제공. 웨딩북은 **실시간 상담·비교견적·후기·즉시예약/잔여타임**. (iwedding.co.kr, weddingbook.com)
- **웨딩홀 핵심 스펙**: **식대(1인당)+대관료+최소보증인원**이 3대 결정요소(보증인원 미달 시에도 그 인원분 식대 결제). 호텔 식대 10만+, 컨벤션 5만+. (ikidari.co.kr, gongysd.com)
- **결제 문화**: 웨딩은 대부분 **계약금(평균 ~300만원, 대관료 50%) + 잔금** 분할. 즉시 전액 온라인결제보다 **상담·예약금** 중심. (threads/계약금, oopy 체크리스트)
- **상담 전환 유도**: 상담예약→계약 시 **캐시백/혜택**(예: 10만원) 제공으로 리드 전환. (weddingbook)
- **결제 수수료**: 카카오페이 온라인 카드 일반 **1.62%**·머니 **1.45%**·영세 **0.38%**. (partner.kakaopay.com, zdnet)
- **마켓 수수료 구조**: 결제수수료 + **판매(중개)수수료 분리**. 네이버 네이버페이 주문관리 1.98%(영세)~3.63%(일반) + 판매수수료 ~2.73%; 카카오 선물하기 판매수수료 ~10%대. (windly.cc, allra.co.kr, ebn)
- **셀러 도구**: 스마트스토어는 **이미지 기반 상세페이지**가 표준, 정산내역에서 **수수료 등급/정산 예상** 확인. (sellerking, naver 판매자센터)

### 9.2 Dewy 적용안 (기획 반영)
1. **상세페이지 핵심 스펙**(§3.1 보강): 카테고리별 결정요소 우선노출 — 웨딩홀=식대/대관료/최소보증인원, 스튜디오=패키지·원본/수정컷·추가금, **공통=추가비용·주차(발렛)·영업시간·위치**. 가격은 "시작가~ + 추가금 명시"로 **투명가격**.
2. **결제 전략 분리**(§3.2/§4 보강): 웨딩 고액 품목(홀·스드메)은 **즉시 전액결제 부적합** → **상담/문의·예약금** 중심. **즉시 카카오페이 결제는 소액·부가 상품/이벤트 참가비**에 적합. → **"결제지원 토글"을 상품별로** 두는 우리 방향이 적절(업체가 즉시결제 적합 상품만 ON). (향후 '예약금 결제' 옵션 검토 항목으로.)
3. **CTA 표준화**: 문의하기(전화/인앱채팅)를 기본, 결제지원 시 구매하기 병행. **비교견적·후기**는 이미 보유(리뷰) — 상담 전환 동선 강화.
4. **혜택→전환**: 쿠폰·이벤트 노출 강화(완료) + 추후 **상담/구매 전환 혜택(캐시백)** 검토.
5. **비용 안내 가이드**(§5 확정): 결제수수료(카카오페이)+VAT만, **판매수수료 0(초기제휴)**, **정산 예상금액 미리보기**(스마트스토어 패턴)를 등록 폼에. 안전 안내치 1.62%+VAT.
6. **(추후)** 스드메 **묶음 견적/장바구니·쿠폰 투명가격**(아이웨딩) — 카테고리 구조화 이후 검토.

## 10. 제약 (반복 명시)
- sandbox: Supabase·카카오 PG·실DB 차단 → **결제·실데이터 e2e 불가**. 마이그레이션·edge function은 **배포(main)에서만** 적용. 정적검증(build·lint·esbuild) + UI e2e(목)까지가 본 환경 한계.

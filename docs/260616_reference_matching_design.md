# 레퍼런스 이미지 매칭 + 속성 필터 — 기능 설계서 (제휴업체 한정)

> 작성 260616. **설계 단계 문서**(구현 전 합의용, 코드 변경 없음).
> 요구: 사용자가 업체를 찾을 때 **레퍼런스 이미지를 등록**하면, 업체 등록 이미지를 기반으로
> **취향에 맞는 제휴업체**를 연결한다. 단,
> ① 네일·헤어·스냅·부케 등 **기타 상세업체까지** 대상에 포함,
> ② 축가처럼 이미지로 못 맞추는 카테고리는 **속성 필터링**으로,
> ③ **레퍼런스(이미지)** 와 **필터** 를 사용자가 **선택**할 수 있게.

---

## 1. 한 줄 요약

"내가 원하는 분위기 사진 1~3장" → (제휴업체 갤러리와 스타일 매칭) → **분위기 비슷한 제휴업체
우선 추천**. 이미지가 의미 없는 카테고리(축가·사회자 등)는 **속성 필터**로 대체하고, 두 방식은
검색 진입 시 사용자가 고른다. **기존 인프라(업로드 버킷·업체 갤러리·제휴 플래그·스타일 태그·
quoteMatch 점수엔진)를 최대한 재사용**하고, 새로 필요한 건 "이미지→스타일 신호" 한 조각뿐.

## 2. 현재 코드 기준 — 이미 있는 것 / 없는 것

**재사용 (이미 있음)**
- 사용자 이미지 업로드: `quote-uploads` 버킷 + `quote_requests.image_paths[]`(최대 3장),
  `QuoteNew.tsx` 업로드 동선, `useQuotes.getPublicUrl` (`20260615043314_quote_request_images.sql`).
- 업체 이미지: `places.main_image_url`(대표 1장) + `place_media`(갤러리 1:N, `kind=photo`,
  `20260521070000_place_media.sql`).
- 제휴 식별·필터: `places.is_partner`·`partner_tier`(basic/friends/bff)·`partner_rank`,
  제휴 한정 필터 이미 동작(`useVenues.ts:40`), 정렬 `partner_rank desc`.
- 스타일 태그(통제 어휘): `collectStyleTags`(`vendorInfoLines.ts:182`) —
  `hall_styles`·`shoot_styles`·`dress_styles`·`makeup_styles`·`hanbok_types` 등 카테고리별 enum.
- 매칭 점수 엔진: `quoteMatch.ts`(지역>예산>파트너>평점 결정적 점수화) — **여기에 차원 추가**.

**없음 (이번에 만들 것)**
- pgvector/임베딩 전무(현 "RAG"는 벡터 아닌 구조화 키워드 주입).
- 비전(분석) 모델 미사용 — 이미지 AI는 전부 *생성*(gpt-image-2). 사진을 *분석/태깅*하는 경로 없음.
- 기타 상세업체(etc) 세분 스타일 태그 — 아래 §3 참조.

## 3. 카테고리 모델 — 시각 vs 비시각, 그리고 'etc' 세분

`places.category` enum 12종(`20260613100000`): wedding_hall·studio·dress_shop·makeup_shop·
hanbok·tailor_shop·honeymoon·appliance·jewelry·invitation_venue·planner·**etc**.
**네일·헤어·스냅·부케·축가·본식DVD·사회자** 등은 전부 `category='etc'` 아래 `places.tags` 로
세분(보드 슬롯은 더 세분 — `vendorBoard.ts:4`). 따라서 상세업체 매칭은 **`etc` + tags 서브타입**을
1급으로 다뤄야 한다.

| 매칭 방식 | 카테고리 | 근거 |
|---|---|---|
| **레퍼런스 이미지** (+필터 가능) | wedding_hall·studio·dress_shop·makeup_shop·hanbok·tailor_shop, **etc:스냅·네일·헤어·부케·본식DVD(썸네일)** | 비주얼 취향이 핵심. 갤러리·스타일 태그 존재 |
| **속성 필터 전용** (이미지 무의미) | **etc:축가·사회자**, planner | 음원/진행은 사진으로 못 맞춤 → 장르·성별·악기·가격·샘플(음원/영상)·경력으로 필터 |
| **혼합 선택** | honeymoon·appliance·jewelry | 제품/지역 스펙 위주지만 비주얼도 일부(주얼리 디자인·허니문 풍경) → 둘 다 제공 |

**핵심 설계 결정**: 매칭 방식은 카테고리 속성(`matchMode: "visual" | "filter" | "both"`)으로
**메타데이터화**한다. 새 상세업체 추가 시 이 한 줄만 지정하면 UI/엔진이 자동 분기(드리프트 방지).

## 3.5 업체 포트폴리오 — 장소·태그·설명 + "같은 장소 우선" (★ 핵심 신호)

업체가 포트폴리오(사진/영상 썸네일)를 등록할 때 **그 작업을 진행한 장소(식장) 정보 + 스타일
태그 + 설명**을 함께 입력하게 한다. 그러면 **사용자가 확정한 식장과 같은 장소에서 진행한
포트폴리오를 가진 업체를 우선 추천/노출**할 수 있다.

> 예: 사용자가 **T웨딩홀**을 식장으로 확정 → T웨딩홀에서 촬영/진행한 포트폴리오가 있는
> **스냅·본식DVD·보정** 업체가 상단에 "T웨딩홀에서 촬영한 포트폴리오 있어요" 배지와 함께 노출.

**왜 강력한가**: 스타일 태그 유사도(자카드)는 근사치지만, **같은 장소 매칭은 정확·결정적·설명
가능**한 신호다. 사용자 입장에선 "내 식장을 잘 아는 업체"라 신뢰가 즉시 선다(동선·조명·구도
파악). 비주얼 임베딩 없이도 Phase 1 에서 바로 가능하고, 효과가 가장 클 차원이다.

**데이터 연결(이미 있음)**:
- 사용자 확정 식장: `user_wedding_settings.wedding_venue_place_id`(+`wedding_venue_name`) —
  `recordVendorDecision`/`SetAsWeddingVenueButton` 가 이미 세팅(이번 감사에서 좌표 시드도 수정).
- 업체 포폴: `place_media`(소유자 쓰기·공개 읽기·검토 면제, `20260521070000`) 를 **확장** —
  현재 `kind(photo/menu)·image_url·title·price·display_order` 에 아래 컬럼 추가.

**place_media 확장(안)** — 별도 테이블 신설 대신 기존 갤러리 확장(DRY):
| 컬럼 | 용도 |
|---|---|
| `venue_place_id uuid references places(place_id)` | 작업 진행 장소(식장)가 Dewy 에 등록돼 있으면 FK. **embed/관계용 FK 필수**(이번 감사 quote_responses embed 교훈) |
| `venue_name text` | 식장이 미등록이면 자유 입력 폴백(표시·부분매칭용) |
| `style_tags text[]` | 카테고리별 통제어휘 스타일 태그(§6 어휘) |
| `description text` | 포트폴리오 설명(검색 보조·노출 문구) |
| `kind` 확장 | `'photo' | 'menu' | 'portfolio'` 또는 photo 에 위 필드 부가 |

**등록 UI**: `BusinessGallery.tsx`(현 갤러리 관리)에 포폴 항목별 **장소 선택(식장 검색 → place_id,
없으면 직접 입력)·스타일 태그·설명** 필드 추가. 운영 검토는 현행처럼 면제하되, 장소 클레임
악용(유명 식장 사칭) 방지를 위해 배지 문구는 "**업체가 등록한** 포트폴리오" 로 정직하게(§7).

**매칭 규칙(요지)**: 후보 업체 중 `place_media.venue_place_id == user.wedding_venue_place_id`
가 1건이라도 있으면 **최상위 가산**(스타일 태그·지역 점수보다 큰 가중). `venue_place_id` 없고
`venue_name` 이 확정 식장명과 정확/부분 일치하면 그다음 가산(label vs value 주의 — §7).

## 3.6 카테고리별 상세 필터 택소노미 (업계 자료 + 패션앱 분석 반영)

요청: 예복·한복 등도 적용하고 **엄청 상세한 필터**가 있으면 좋겠다. → 업계 자료 + 한국 패션앱
(무신사·29CM·지그재그·에이블리) 필터 체계를 조사·분석해 카테고리별 **패싯(facet) 필터**를 설계.

**패션앱에서 가져온 원칙**
- **패싯 다중선택 + 결과 수 표기**(무신사식): 각 필터값 옆 "N개". 빈 결과 조합은 비활성/안내(dead-end 금지).
- **공통 패싯**(전 카테고리): 가격대·지역·제휴여부·**서비스형태(기성/맞춤/대여)**·브랜드/제작처·색상.
- **실측 필터**(무신사 고유): 사이즈를 라벨(S/M/L) 대신 **숫자 실측**(총장·어깨·가슴단면·소매)으로 →
  예복/드레스/한복 사이즈 매칭에 차용(택일 가능).
- **무드/스타일 패싯** = 레퍼런스 이미지 매칭의 태그와 **동일 어휘**(§핵심: DRY).

**★ 핵심 설계 — 단일 어휘 3중 사용(DRY)**: 아래 통제 어휘 한 벌이
① 속성 필터값 ② 이미지→스타일 태그 분류 결과(§6) ③ 업체 포폴 태그(§3.5) 에 **모두 동일하게** 쓰임.
어휘는 `src/lib/categoryFacets.ts`(신규) 단일 소스 — 기존 `categoryLabels`·`vendorInfoLines` 패턴.
스키마는 기존 `place_*_shops` 의 단일 `*_styles text[]` 를 **패싯별 컬럼**으로 확장(`*_styles` 는 무드 유지).

### 가치(values) 패싯 — ★ 이미지로 못 잡는 축은 기존 자산 재사용 (DRY)
친환경·비건·반려동물 동반·종교·**임산부 편의**·영문안내 같은 축은 사진으로 분류 불가(이미지
매칭의 사각). 이건 **속성 값**으로 다루되, 앱에 **이미 구현된** 가치필터를 재사용한다(중복 금지):
- 단일 소스 `src/lib/placeValueTags.ts`(`PLACE_VALUE_TAG_OPTIONS`), 상태 `useFilterStore.valueTags`,
  쿼리 `useVenues` 의 `.overlaps`, **가용성 가드 `useValueTagAvailability`(0건 칩 자동 숨김 = dead-end 방지)**,
  UI `ValueTagModal`/`ValueTagChipRow`. (persona-sim v2 권고 #7 산출물.)
- 매칭 엔진은 후보 필터에 `valueTags.overlaps` 를 더하고, 패싯 UI 는 위 컴포넌트를 그대로 쓴다.
  **임산부 편의는 여기에 값 추가**(드레스 임부 가능·메이크업 약품 안내 등)로 확장(F-1).
- 신규 패싯(예복·한복·드레스 등)도 **`useValueTagAvailability` 식 가용성 가드 패턴을 동일 적용**해
  0건 필터값은 자동 비활성/숨김(빈 결과 노출 금지).

## 4. 사용자 선택 UX (레퍼런스 ↔ 필터)업체 둘러보기/견적 진입 시 상단에 **세그먼트 토글**:
- `🖼 레퍼런스로 찾기` — 사진 1~3장 업로드 → "이 분위기와 비슷한 제휴업체"
- `⚙️ 조건으로 찾기` — 기존 필터(지역·예산) + 카테고리별 속성 필터(축가: 장르/성별/악기 등)

규칙:
- `matchMode="filter"` 카테고리(축가 등)는 **레퍼런스 토글 비활성**(설명 툴팁: "이 카테고리는 조건
  검색이 더 정확해요") — *dead-end 토글로 두지 않는다*.
- `matchMode="both"` 는 둘 다, 사용자가 전환. 기본값은 카테고리별로 더 적합한 쪽.
- 레퍼런스/필터는 **상호 배타가 아니라 가산** 가능: 레퍼런스 + 지역/예산 필터 동시 적용(이미지
  점수 위에 기존 region/budget 점수 합산).

## 4.1 "아직 모르겠어요" — 초보자용 예시 레퍼런스 (취향 진단) ★

준비가 처음이라 **취향 자체를 모르는 사용자**는 레퍼런스 사진을 올릴 수도, 상세 필터를 고를 수도
없다(빈 입력 → dead-end). 그래서 레퍼런스 단계에 **`🤍 아직 모르겠어요`** 선택지를 둔다. 누르면
**큐레이션된 예시 레퍼런스 보드**(스타일별 대표 이미지 묶음)를 보여주고, 사용자는 "끌리는 것"만
탭하면 된다. 이게 곧 취향 진단이 되어 동일 엔진으로 흘러간다.

**전역 무드 1회 진단 → 카테고리 상속 (페르소나 시뮬 X3 반영)**
취향을 카테고리마다 다시 고르게 하면 초보·시간압박(F-1)이 이탈한다. 그래서 **전역 "웨딩 무드"
진단을 1회**(예: 로맨틱·모던·클래식·전통·미니멀) 받고, 각 카테고리는 이를 **기본값으로 상속**한
뒤 필요 시 카테고리별로만 미세조정한다. designer/효율형(재혼 G-1 등)은 **진단을 스킵**하고 바로
필터/업로드(planning_style·페르소나 분기).

**동작 (Pinterest/패션앱 온보딩식 "취향 고르기")**
1. 카테고리별 예시 보드 노출 — 드레스: `클래식·모던·로맨틱·볼드·빈티지`, 예복: `클래식 턱시도·
   모던 슬림·캐주얼 셋업`, 한복: `전통·퓨전·드레스한복` 각 대표 이미지 2~3장.
   (혼주용은 `target` 패싯으로 분리 — S-1 부모 보수 취향.)
2. 사용자가 마음에 드는 카드 **다중 선택**(또는 "이 중엔 없어요").
3. 선택 → **§3.6 통제 어휘 태그로 환산**(DRY: 필터·이미지매칭·포폴과 같은 어휘) →
   `user_taste_profiles` 에 저장 → 그대로 매칭 엔진(`styleMatchScore`)에 투입.
4. 결과와 함께 "**고르신 분위기**: 로맨틱·내추럴" 를 보여줘 사용자가 **자기 취향을 학습**(교육 효과).

**핵심 — 예시는 실제 제휴업체 포트폴리오에서 큐레이션** (dead-end 금지 + 일석이조)
- 예시 카드를 가상 이미지가 아니라 **고품질·태그된 제휴업체 포폴(`place_media`)에서 추출**하면,
  사용자가 고른 예시가 **곧 실제 추천 가능한 업체**로 직결된다("이 분위기 = 이 업체"). 장식용
  플레이스홀더가 아니라 실 전환 경로(검증 규칙의 dead-end 회피).
- 운영자/제휴업체가 "대표작" 플래그를 단 포폴을 풀로 사용(없으면 평점·완성도 상위 폴백).

**세그먼트 연결**: 이미 있는 `user_wedding_settings.planning_style='beginner'` 와 페르소나(초보)
사용자에겐 이 "아직 모르겠어요 → 예시 진단"을 **기본 경로로** 먼저 제안(상세 필터는 접어둠).
숙련/디자이너 성향(`designer`)에겐 상세 필터·직접 업로드를 먼저.

**경량 버전(이미지 진단이 부담일 때)**: 예시 보드 대신 2~3문항 미니 퀴즈("화려 vs 심플", "전통
vs 모던")로도 같은 태그를 도출 가능 — 비전 호출 0, Phase 0 에서 즉시 가능.

## 6. 구현 단계 (벡터부터 가지 말 것)

**Phase 1 — 비전→스타일태그 분류 매칭 (권장 시작)**
- 비전 모델(Claude/Gemini vision)이 레퍼런스 사진을 **기존 통제어휘 스타일 태그**로 분류.
- 업체 스타일 태그(`collectStyleTags`)와 **집합 유사도(자카드/가중)** 매칭 → `quoteMatch` 점수 차원.
- 장점: **설명 가능**(어떤 태그가 맞아서 추천), 저비용·저지연, pgvector 운영 0, 기존 메타데이터 재사용(DRY).
- etc 상세업체용 스타일 어휘 신설 필요(스냅: 필름/디지털/내추럴/감성/야외, 네일: 심플/글리터/프렌치…,
  부케: 생화/조화/내추럴/클래식, 헤어: 업스타일/다운/한복헤어…). 업체 등록 폼에도 태그 입력 추가.

**Phase 2 — 진짜 이미지 임베딩 (효과 검증 후)**
- pgvector + 업체 갤러리 임베딩 사전계산(비동기 edge function) + HNSW 코사인.
- 미묘한 비주얼 유사도까지. 단 운영·비용·콜드스타트 부담 → Phase1로 수요/정확도 검증 후 결정.
- Phase1 태그 점수와 **블렌딩**(설명가능성 유지) 가능.

## 7. 반드시 처리할 함정 (AGENTS 규칙)

1. **제휴 한정 = 작은 풀 → 빈 추천(dead-end) 금지 [폴백 기본 ON]**: 지방·니치(부산 G-2·청주
   한옥 S-1)는 제휴 풀이 구조적으로 작아 유사 0건이 흔하다. 폴백을 옵션이 아니라 **기본 동작**으로
   못박는다 — 제휴 우선 + 부족분 비제휴 보충 + "이 조건의 제휴가 적어 전체에서 보여드려요" 정직
   안내. placeholder/빈 결과를 "추천함"으로 두지 않는다(검증 규칙).
   - **같은-식장 포폴은 식장이 places 에 등록돼야 강함**(X5). 지방·한옥·소규모는 미등록 多 →
     `venue_name` 텍스트 폴백 비중을 높이고 식장 등록을 유도.
1b. **예시 풀·어휘의 중앙 편향 가드(X4)**: 예시 진단 이미지가 서울·초혼·20대·정형체형으로 쏠리면
   재혼(G-1)·임부(F-1)·전통(S-1)이 또 소외(v2 결론 반복). 예시 샘플의 연령/재혼/체형/전통 다양성과
   어휘 포함성(혼주 `target`·임부 편의·전통)을 가드. 카테고리 데이터 부족 시 경량 퀴즈로 폴백.
2. **콜드스타트**: 제휴업체 다수가 갤러리 빈약(대표 1장만) → 매칭 근거 부족. 업체 사진/태그 등록
   유도(대시보드 넛지) 선행. 태그 0개 업체는 매칭 제외하되 기존 정렬로 노출(회귀 없음).
3. **카테고리 혼선**: 식장 사진 ≠ 드레스 취향. 레퍼런스는 **검색 중인 카테고리 컨텍스트**로 해석
   (드레스 탭의 레퍼런스는 dress_styles 로만 태깅).
4. **설명가능성 필수**: "AI 추천"만 X. "필름·내추럴 분위기가 비슷해요" 칩(#308 `예산 내`/`지역 일치`
   칩과 동일 패턴) — 신뢰 + 사용자 학습.
5. **garbage-in**: 무관 사진(음식 등)·저품질 → 신뢰도 낮으면 매칭 미적용하고 필터로 안내(조용한 오작동 X).
6. **비용/지연**: 비전 호출은 업로드당 1회 + 결과 캐시(같은 사진 재분석 금지). 업체 측은 등록 시 1회 사전계산.
7. **프라이버시**: 레퍼런스 사진은 사용자 소유 폴더(RLS), 매칭 결과만 노출. 타 사용자 사진 접근 경로 0.

# 푸시 알림 발송 시나리오 정의

출시 체크리스트 5번("기능 붙이기 전에 언제 보낼지부터 정한다") 대응 문서.
**1차 출시(v1)에서는 푸시 미활성** — Firebase 미설정 + 사용자 토큰 등록 미배포.
아래는 활성화 시 적용할 발송 시나리오를 사전 확정한 것이다.

## 현재 상태

- `supabase/functions/send-push/index.ts`: FCM HTTP v1 발송 함수 구현 완료(호출처 없음).
- `supabase/migrations/...device_tokens.sql`: 디바이스 토큰 테이블(미배포).
- 클라이언트 토큰 등록(@capacitor/push-notifications) 미구현.
- 결론: **플러밍 일부 존재, 실제 발송 트리거 0개.** 활성화 선행조건은 Firebase 프로젝트 설정 + 토큰 등록 배포.

## 발송 원칙

1. **사용자가 켠 알림만** 발송한다(설정 화면 토글 + `user_consents`/알림 설정 존중).
2. **빈도 상한**: 1인당 하루 최대 2~3건. 마케팅성은 주 1회 이하.
3. **딥링크 필수**: 모든 알림은 관련 화면으로 바로 이동.
4. **조용한 시간**: 21:00~08:00 발송 금지(예약은 다음 날 오전).

## 핵심 시나리오 (활성화 시 우선 3종)

| # | 시나리오 | 트리거 | 발송 시점 | 딥링크 | 동의 카테고리 |
|---|---|---|---|---|---|
| 1 | **D-Day/할 일 리마인더** | 예식일 기준 마일스톤(D-180/90/30/7) 또는 시기별 골든타임 | 오전 9~10시 (pg_cron 일배치) | `/schedule` | 일정 알림(서비스성) |
| 2 | **커뮤니티 내 글 반응** | 내 게시글에 댓글/답글 작성 | 발생 즉시(조용한 시간 제외) | `/community/:id` | 활동 알림(서비스성) |
| 3 | **쿠폰·이벤트 안내** | 찜한 업체의 신규 쿠폰/이벤트 승인 노출 | 오전 11시 배치 | `/coupons` 또는 업체 상세 | 마케팅 알림(선택 동의) |

> 1·2번은 서비스성(이용 관련) 알림으로 기본 on, 3번은 마케팅성으로 **명시적 동의(opt-in)** 필요. 마케팅 동의는 `user_consents`(marketing_v1)와 동기화한다.

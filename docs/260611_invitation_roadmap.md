> 진단 근거: 코드 갭 분석(스튜디오/관리자/뷰어) + 경쟁 분석(docs/260611_competitor_analysis.md).
> 핵심 발견 — **AdminTemplateEditor 에는 드래그·리사이즈·그리드 스냅·에셋 배치가 이미 다 있다**
> (`src/pages/admin/AdminTemplateEditor.tsx:107-213`). 사용자 스튜디오만 "숫자 입력 편집(V1)"에

## Phase 2 — 인터랙션 모바일 청첩장 (경쟁사 표준 기능)

경쟁사(웨딩북 무제한 무료 청첩장, 스냅포스트 등) 기준 "있어야 기본"인 것:

| # | 기능 | 설계 포인트 | 규모 |
|---|---|---|---|
| 2-1 | **포토 갤러리/라이트박스** | 새 슬롯 타입 `gallery` (이미지 N장 + 스와이프). 업로드는 기존 invitation-uploads 재사용 | 중 |
| 2-2 | **방명록(축하 메시지 공개 벽)** | `invitation_guestbook` 테이블 — invitation_rsvp 와 동일 RLS 패턴(발행본만 INSERT, 작성자 표시 공개 SELECT, 행 상한 트리거). RSVP 메시지와 구분 | 중 |
| 2-3 | **지도 내비 연동** | map 액션 탭 시 시트: 카카오맵/네이버지도/티맵 딥링크 (스킴 URL, 설치 미존재 시 웹 폴백) | 소 |
| 2-4 | **카카오톡 공유 썸네일(OG)** | `/i/:slug` SSR 메타태그 — `api/_lib`(ssr) 인프라 재사용. 대표사진+신랑신부 이름 OG | 중 |
| 2-5 | **계좌 아코디언** | 양가 각 다계좌(부모님 포함) 그룹 펼침 + 복사. user_data 스키마 확장 | 소 |
| 2-6 | **배경음악** | 토글 가능한 오디오 플레이어, 자동재생 정책 대응(첫 터치 후 재생) | 소(후순위) |

## Phase 3 — 청첩장 외 경쟁 격차 (경쟁분석 P1, 청첩장 안정화 후)

- D-365 단계별 타임라인 개인화(요즘웨딩 코어 UX, 페르소나 엔진 활용)
- 이벤트/혜택 탭 벤더 제휴 채널화(웨딩북Biz 라이트)
- 발행/생성 성공 모먼트 리뷰 유도 + 포인트 보상
- 축의금 펀딩·카드결제는 중장기 검토만(PG/정산 부담, 웨딩북 선점)

## 진행 원칙

- 순서: 1-1 → 1-2 → 1-3 (편집 가능 상태 확보) → 1-4·1-5 → Phase 2 는 2-3·2-5(소) 먼저, 2-1·2-2 본격.
- 각 단계마다: vitest + build + 실기기 발행→뷰어 e2e. DB 변경은 멱등 마이그레이션 + MCP 적용 + 롤백 트랜잭션 검증(verification-lessons 준수).
- 매 단계 별도 커밋 — 중간에 끊겨도 main 은 항상 동작 상태.

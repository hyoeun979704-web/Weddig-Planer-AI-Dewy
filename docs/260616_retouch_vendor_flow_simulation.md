# 사설보정 제휴업체 — 홈→계약→결과물 수령 플로우 기획 & 시뮬레이션 (화면녹화용)

> 목적: 사설보정(사진 후보정) **제휴업체와 함께하는 화면녹화 시뮬레이션** 대본. 현재 코드 기준
> (커밋 `130f9214`, 260616)으로 홈 진입 → 발견 → 문의/상담 → 계약 → 결과물 수령까지 **실제로
> 보여줄 수 있는 화면**과 **인앱에 없어서 오프앱으로 처리/내레이션할 부분**을 분리한다.
> 검증 규칙: "동작한다 ≠ 검증됨" — 각 단계는 코드 경로로 확인, 막힘은 ❌ 로 정직 표기.

## 0. 핵심 결론 (녹화 전 반드시 인지)

- **보정은 `places.category='etc'` 상세업체** — 구조화 견적(QuoteNew/`create_quote_request`)의
  대상 카테고리가 **아니다**(quoteCategory: wedding_hall·studio·dress·makeup·suit·hanbok 만).
  → 보정은 **"예약 문의"(인앱 문의)** 경로로 상담·계약 협의한다(견적 매칭 아님).
- **인앱 문의는 진짜 동작**: 소비자 상세 → "예약 문의" → `place_inquiries` insert →
  사장님 `BusinessInquiries`("문의 관리")에서 답변. RLS 로 내 업체 문의만. ✅
- **계약/결제·결과물(보정본) 수령은 인앱 기능이 없다** → 오프앱(문의 협의·외부 전달).
  녹화에서 이 두 구간은 **내레이션 + 오프앱 화면**으로 메우거나, 후속 구현 후 재촬영.

## 1. 녹화 전 셋업 체크리스트 (카메라 앞에서 안 막히려면)

제휴업체 계정·데이터가 아래를 만족해야 인앱 경로가 매끄럽다(미충족 시 dead-end 토스트 발생):
- [ ] 업체가 **입점(claim) + 승인** 상태(`is_partner`/claimed). 미입점이면 "문의하기"가
      연락처 폴백/토스트로 빠짐(`PlaceDetailLayout.tsx:317`).
- [ ] 문의 채널 = **chat(인앱)** 로 설정(사장님). url/phone 이면 외부로 튕겨 녹화 흐름 끊김.
- [ ] **포트폴리오 갤러리**(`place_media`) 등록 — 보정 전/후 샘플(상세페이지 볼거리).
- [ ] 대표 이미지·지역·태그("보정"·"사진보정") 입력 → 검색/둘러보기 노출.
- [ ] 사장님 단말 1대 더(또는 분할화면) — 소비자↔사장님 양쪽을 한 녹화에 담으려면.
- [ ] (선택) `place.quote_request_url` 설정 시 상세에 "견적 요청하기 →" 외부링크 노출.

## 2. 녹화 시나리오 (씬별 — 소비자 시점 기준)

| # | 씬 | 화면/경로 | 보여줄 것 | 상태 |
|---|---|---|---|---|
| S1 | 홈 진입 | `/` (Home·PersonaDashboard) | D-day·진행률·추천. "보정 업체 찾기" 의도 제시 | ✅ |
| S2 | 발견 | 검색 "보정" → `/results` **또는** `/vendors/기타` 둘러보기 **또는** `/board` | 보정 업체 카드 노출 | ⚠️ 보정 전용 동선 약함(기타 묶임) |
| S3 | 업체 상세 | `/vendor/:id` (VendorDetailPage) | 포폴 갤러리(보정 전/후)·평점·소개·제휴 배지 | ✅ (갤러리 등록 시) |
| S4 | 문의 시작 | 상세 하단 "**예약 문의**" → `PlaceInquirySheet` | 제목·내용·연락처 입력 → "문의를 보냈어요" | ✅ (입점+chat 채널) |
| S5 | 사장님 응답 | 사장님 `/business/inquiries` (문의 관리) | 도착 문의 → 답변 등록(status=answered) | ✅ |
| S6 | 상담·계약 협의 | (문의 답변 내 / 오프앱) | 일정·가격·보정 범위 협의 | ⚠️ 인앱 계약서·결제 없음 → 협의/오프앱 |
| S7 | 계약 확정 표시 | (인앱 상태 없음) — 보드 슬롯 "예약완료"는 견적 경로 전용 | 보정은 예약완료 표시 경로 없음 | ❌ 인앱 미지원 |
| S8 | 결과물(보정본) 수령 | (인앱 없음) — 이메일/클라우드 링크 | 외부 전달 화면 | ❌ 인앱 미지원 |

> S1~S5 는 인앱으로 **완전 녹화 가능**. S6~S8 은 현재 인앱에 없어 **오프앱/내레이션**으로 처리.

## 3. 시뮬레이션 — 단계별 마찰 (✅/⚠️/❌)

- **S2 발견 ⚠️**: 보정은 "기타"에 섞여 전용 필터·타일이 없음. 녹화에선 **검색 "보정"** 또는
  업체 직링크(`/vendor/:id`)로 바로 진입하는 게 깔끔. (개선: 보정 태그 필터/타일 — 설계 §3.6.)
- **S4 문의 ✅(조건부)**: 입점+chat 이면 인앱 문의 시트가 실제 insert. **미입점이면 토스트
  폴백**(dead-end) — 셋업 체크리스트 필수.
- **S6 계약 ⚠️**: 인앱 계약서/계약금 결제 없음. 문의 스레드 내 텍스트 협의뿐(파일첨부 없음).
- **S7 예약확정 ❌**: 견적(quote) 경로의 "예약 완료로 표시"·보드 자동반영은 **보정엔 적용 안 됨**
  (quoteCategory 부재). 보정 계약을 인앱에 "확정"으로 남길 데가 없음.
- **S8 결과물 수령 ❌**: 보정본 파일을 인앱에서 받을 방법이 전무(문의/스레드 모두 텍스트만).
  → 사장님이 이메일·드라이브 링크로 전달, 소비자 수동 다운로드.

## 4. 권고 — "완결형 보정 플로우"를 인앱으로 만들려면 (후속 구현)

녹화에서 드러난 dead-end(S7·S8)를 메우는 최소 구현 순서:
1. **보정 발견성**(S2): `/vendors/기타` 에 보정 태그 칩 필터(설계 §3.6 패싯). [低]
2. **보정 계약 상태**(S7): etc 상세에 "상담 예약/계약 확정" 경량 상태(문의→예약 전환) 또는
   보정용 quoteCategory 신설. [中]
3. **결과물 수령**(S8): 사장님→소비자 **파일 전달**(스토리지 버킷 + 다운로드/리뷰요청).
   문의 스레드 파일첨부 또는 별도 '내 보정본' 페이지. [中~高, 라이브 DB 검증 필요]
4. **같은-식장 포폴 우선**(이미 PR #314 기반): 보정 업체가 내 식장 본식을 보정한 포폴 있으면
   우선 노출 → 신뢰. 보정에 특히 강력(동선·조명 이해).

## 5. 한 줄

> **S1~S5(홈→발견→상세→문의→사장님 응답)는 인앱으로 매끄럽게 녹화 가능**(입점+chat 셋업 전제).
> **S6~S8(계약·결제·보정본 수령)은 인앱 기능이 없어** 오프앱/내레이션으로 처리하거나, §4 후속
> 구현 후 재촬영을 권장. 녹화 전 §1 체크리스트로 dead-end 토스트부터 차단할 것.

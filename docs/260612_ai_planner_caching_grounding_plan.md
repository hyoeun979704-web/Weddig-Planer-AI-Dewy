## Part A. 프롬프트 캐싱 + 2-tier (비용 절감)

### A1. 캐싱 원리 (OpenAI automatic prompt caching)
OpenAI 는 1024토큰↑ 프롬프트의 **공통 접두(prefix)** 를 자동 캐싱(5~10분 TTL), 캐시 적중분 입력비 ~50%↓·지연↓. 핵심은 **정적 접두 고정 + 동적 꼬리 분리**.

현재 시스템 프롬프트 조립순서(index.ts:`systemPrompt`):
```
BASE_SYSTEM_PROMPT + ALWAYS_ON_CAPSULES + conditionalCapsules + userContext
```
- 앞 2개(BASE + 부모설득 캡슐)는 **모든 요청 공통 = 캐시 대상**.
- 뒤 2개(페르소나별 캡슐 + 사용자 컨텍스트)는 **요청마다 달라 캐시를 깨뜨림**.

### A2. 변경: 정적/동적 메시지 분리
시스템 프롬프트를 **2개 메시지**로 쪼갠다 (OpenAI messages 배열은 prefix 캐싱):
```
messages = [
  { role: "system", content: STATIC_PROMPT },      // BASE + ALWAYS_ON — 캐시되는 고정 접두
  { role: "system", content: DYNAMIC_CONTEXT },    // conditionalCapsules + userContext (사용자별)
  ...conversation,
]
```
- STATIC_PROMPT 는 코드 상수라 모든 사용자·요청 동일 → 캐시 적중률 최대화.
- DYNAMIC_CONTEXT 가 바뀌어도 첫 system 메시지까지는 캐시 유지.
- 효과: 정적 프롬프트(~3~5k토큰)가 입력의 대부분 → 캐시 적중 시 입력비 대략 반감(검토 비용표의 캐싱 적용분).

### A3. 2-tier 라우팅 (gpt-4o-mini 기본 / gpt-4o 승격)
모델 선택을 edge function 에서 결정:
- **기본 gpt-4o-mini**: 대부분 LLM 라우팅 턴.
- **gpt-4o 승격 조건**(아래 중 하나):
  1. 직전 사용자 메시지가 **문제해결·비교·협상·갈등** 신호 (intentRouter 의 매칭 없음 + 키워드: "비교/고민/설득/어떡하/막막")
  2. **대화 길이** ≥ 6턴 (맥락 누적 — 일관성 요구↑)
  3. **고난도 페르소나**: weddingSettings 의 pregnant / marital_history=remarriage / 부모부재 (도메인 캡슐 주입된 케이스)
- 신호는 클라이언트(useAIPlanner)가 `tier_hint` 로 보내거나, 서버가 마지막 메시지+컨텍스트로 판정. 서버 판정이 단일 소스라 권장.
- DB 즉답(intentRouter dbHandler 매칭)은 **LLM 미호출** — 기존 0환각·0비용 경로 그대로(가장 큰 비용 절감은 여기).

### A4. 측정·가드
- 응답 헤더에 `X-Model-Tier`, `usage`(캐시 적중 토큰 포함) 로깅 → 일별 모델별 토큰·비용 집계.
- 무료 5질문/일 게이트 유지(폭주 차단). 프리미엄도 분당 rate-limit 고려.

## Part B. 환각 최소화 레이어 (신뢰성)

버티컬 챗봇 표준(RAG)·검토 ④신뢰성 갭 대응. **5겹 방어** — 위에서 막을수록 싸고 확실.

### L1. 결정적 우선 (이미 존재 — 커버리지 확대)
intentRouter → DB 핸들러가 사실/개인 데이터(D-day·예산·일정·체크리스트)를 **LLM 없이** 답함 = 구조적 0환각. 
- **갭(테스트로 고정됨)**: 자연어 체크리스트/예산 질문은 키워드 라우터에 안 잡혀 LLM 으로 샘 → 지어낼 위험. 
- **작업**: 라우터 패턴 확대 + "내 데이터" 질문은 LLM 폴백 전에 한 번 더 DB 의도 분류.

### L2. 근거 주입 (RAG) — 가격·업체는 실데이터만
LLM 이 답하기 전, 질문 유형별로 **앱 내 실데이터를 프롬프트에 근거로 주입**하고 그 밖의 수치/업체명 생성을 금지:
- **가격/시세 질문**: `getRegionalAvgWithMeal(region, guestCount)` + `RegionalAverage`(venue·sdm·per_guest_meal 등, src/data/budgetData.ts)를 "근거 데이터" 블록으로 주입. 프롬프트 규칙: *이 표의 숫자만 인용, 없는 항목은 "데이터 없음"*.
- **업체 추천/검색**: places DB 를 지역·카테고리·조건으로 조회해 실제 N건을 주입. *주입 목록 밖 업체명 생성 금지*. (③a 업체검색 RAG 와 동일 인프라 — 함께 구현)
- 주입 데이터가 없으면 L3로.

### L3. 불확실성 계약 (프롬프트 강화 — 구조화)
근거가 없을 때의 행동을 명문화(현 면책 문구를 규칙으로 승격):
- 모든 금액/통계엔 **출처 태그**(`[앱 데이터]` / `[일반 시세 추정]`) + 범위 표기 강제.
- 근거 없으면 **단정 금지, "정확한 수치는 업체별로 달라요. 견적 받아보시는 게 정확해요"** 로 회피가 아닌 정직.
- 법률/계약/의료는 근거 출처(소비자분쟁해결기준·식약처 등) 언급 + "확정 자문 아님" 한 줄.

### L4. 출력 후처리 점검 (값싼 휴리스틱)
응답 생성 후 정규식 스캔:
- 구체 금액(`\d+만원`, `\d+억`) 이 **근거 주입 없이** 등장 → 자동 면책 문구 덧붙임 또는 tier-2 재생성.
- 주입 목록에 없는 업체명 패턴 → 경고 로깅(초기엔 차단 대신 모니터링).
- 비용 0(문자열 검사). 효과 큰 마지막 그물.

### L5. 메모리 검증 (환각 누적 차단)
user_ai_memory 자동 추출이 틀리면 다음 대화까지 오염. 
- 저장 전/후 사용자에게 **확인 칩**("이렇게 기억할게요: '예산 4천만' ✓/✗") → 오인 사실 정정·삭제.
- 메모리 관리 UI(조회/삭제)로 사용자 통제권.

## 구현 순서 & 파일

1. **B-L2/L3 + A2 캐싱 분리 + OpenAI 전환** (한 PR — 같은 프롬프트 조립부): `supabase/functions/ai-planner/index.ts`(메시지 분리·모델 스위치·근거 주입 훅), `prompt.ts`(불확실성 계약 규칙), 신규 `grounding.ts`(가격=budgetData / 업체=places 조회). GEMINI→OpenAI 기본 전환.
2. **A3 2-tier 라우팅**: index.ts 승격 판정 + `intentRouter` 신호 재사용.
3. **B-L1 라우터 커버리지 확대**: `src/lib/chatbot/intentRouter.ts` + 회귀 테스트(이미 고정한 자연어 체크리스트 기대를 'checklist 라우팅'으로 전환).
4. **B-L4 후처리 점검**: index.ts 응답 후 스캔(스트리밍이면 버퍼 후 검사 또는 비스트리밍 tier-2만).
5. **B-L5 메모리 검증 UI**: `src/pages/AIPlanner.tsx` 확인 칩 + 관리 화면.

## 모델·비용 메모 (실측 반영)
eval 21시나리오 usage 측정치: 입력 ~1,700토큰/콜(컨텍스트 빈 eval 계정 기준 — 실사용은 +컨텍스트), 출력 mini 174 / 4o 232 토큰.

- **캐싱 효과**: 입력의 대부분이 정적 시스템 프롬프트(~1.3k/1.7k) → A2 분리로 캐시 적중 시 입력비 ~50%↓ → mini ~$0.30, 4o ~$4 수준.

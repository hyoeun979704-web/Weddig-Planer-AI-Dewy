# 업체 상세페이지 — 전 카테고리 구조 감사 (260619)

> 목적: "기본정보가 카테고리별 구조가 없어 데이터 정리가 안 됨 + 상세페이지가 카테고리마다
> 들쭉날쭉"을 전수 점검하고, **기본정보 구조화 + 데이터 정리** 구현 계획을 합의하기 위한 문서.
> 코드 기준(현 렌더 로직) 감사. 대상: `src/pages/VendorDetailPage.tsx`(+`CategoryExtras`),
> `src/components/detail/PlaceDetailLayout.tsx`(탭/기본정보), `src/components/place/PlaceBusinessSections.tsx`.

## TL;DR

1. **기본정보 탭은 전 카테고리 공통 shell**(이름·평점·주소·태그·갤러리·혜택·연락/위치·CTA). **카테고리별 핵심 스펙(가격·식대·원본컷·캐럿 등)은 전부 '상세정보' 탭에만** 있다 → 첫 화면(기본정보)에서 비교 판단이 안 됨 = "기본정보 구조 없음".
2. **상세정보(CategoryExtras)는 카테고리별 풍부도 격차가 큼**: 웨딩홀·스튜디오·허니문·혼수가전·예물=풍부 / 드레스·청첩장모임=보통 / **메이크업·한복·예복=빈약(필드 자체가 적음)** / **기타(etc)=없음 → 폴백 추가됨(#362)**.
3. **업체 소개(`place.description`)가 어느 탭에도 안 보였다** — 전 카테고리 공통 누락(특히 스키마 없는 기타에 치명적). #362에서 기타 폴백에만 노출. **전 카테고리 노출 권장**.
4. #362에서 이미 처리: 상품 클릭 상세·포트폴리오 라이트박스, 쿠폰·이벤트 기본탭 노출, 기타 폴백.

## 현재 구조

- 탭: **기본정보 / 상세정보 / 리뷰** (`PlaceDetailLayout` `TabKey`).
- 기본정보(`BasicTab`): 갤러리·이름·평점·카테고리·주소·태그 → **혜택(쿠폰·이벤트)** → 연락/영업시간/위치/SNS. (카테고리 무관 동일)
- 상세정보(`DetailTab`): `PlaceBusinessSections`(포트폴리오·상품) + **`CategoryExtras`(카테고리별 스펙)** + 관련 커뮤니티.
- 데이터 소스: `usePlaceDetail` → `places`(+ 카테고리별 컬럼) → `LegacyDetail`.

## 공통 발견(전 카테고리)

- **A. 기본정보에 카테고리별 핵심 스펙 부재** — 가격(시작가)·핵심 1~3개가 상세 탭에만. 첫 화면 비교·전환에 불리.
- **B. 업체 소개(description) 노출 위치 없음** — 전 카테고리 공통 누락(기타만 #362 폴백).
- **C. 얇은 카테고리(메이크업·한복·예복) 데이터 모델 빈약** — 보여줄 필드 자체가 적음 → 입력폼/스키마 보강 필요(제품 결정).
- **D. 가격 표기 일관성** — 상세 대표가는 '상품 관리' 패키지에서 옴(없으면 '문의'). 기본정보에 '시작가/최저가'를 카테고리 공통으로 노출하면 정리됨.

## 권장 구현 계획 (단계)

**Phase 1 — 기본정보 '핵심 스펙 요약' 블록(공통 컴포넌트 1개 + 카테고리별 필드맵)**
- `PlaceKeyFacts` 컴포넌트: 카테고리→`[{label, value}]` 맵으로 위 표의 **핵심 스펙**을 기본정보 탭 상단(혜택 위/아래)에 칩·스탯으로 노출. 값 없으면 자동 생략(빈칸 방지).
- 가격(시작가/최저가)을 카테고리 공통 1행으로 표준화.
- 효과: 카테고리마다 "데이터 정리된" 일관 구조 + 첫 화면 비교 가능.

**Phase 2 — 업체 소개 전 카테고리 노출**
- `place.description`을 기본정보(또는 상세 상단)에 "소개" 섹션으로 공통 노출(현재 누락).

**Phase 3 — 얇은 카테고리 데이터 보강(제품 결정 필요)**
- 메이크업·한복·예복의 상세 입력 필드 확장(스키마/`BusinessVendorEdit` 폼) — 무엇을 받을지 기획 확정 후.

**Phase 4 — 데이터 정리/검수 가이드**
- 카테고리별 필수 핵심 필드 입력 유도(업체 등록 폼 안내) + 운영자 검수 체크리스트.

## 결정 필요(사용자)

- Phase 1의 카테고리별 **핵심 스펙 셋**(위 표 제안) 확정 — 그대로 갈지/가감할지.
- Phase 3 얇은 카테고리에서 **추가로 받을 항목** 정의.
- 우선순위(전 카테고리 일괄 vs 핵심 카테고리 먼저).

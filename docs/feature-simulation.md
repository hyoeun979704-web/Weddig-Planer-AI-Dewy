## 3. 기능별 페인포인트 & 개선 (요약)

> 이 PR에서는 (T) 표시된 튜토리얼 항목만 실제 구현. 나머지는 백로그.

### 3.1 튜토리얼 (T·이번 PR 본 영역)
- **기존(개선 전) 페인포인트** — 아래 "개선(이번 PR)"에서 해소됨(셀렉터 현황: AIPlanner에 `ai-header`/`ai-suggestions`/`ai-input` 존재, 잘못된 `nav-ai` 단계 제거 완료):
  - `useTutorial.ts`의 `APP_TOUR_STEPS`가 `nav-ai` 셀렉터를 참조하지만 BottomNav에는 AI 탭이 없음 → 빈 강조 박스, 툴팁만 떠 있음.
  - AIPlanner 페이지의 가이드는 `ai-header`/`ai-suggestions`/`ai-input` 셀렉터를 참조하지만 페이지에 해당 attr이 **하나도 없음** → 3단계 모두 빈 화면 코치마크.
  - 가이드 목록이 평평한 리스트 (`FEATURE_GUIDES`). 어디서 시작해야 하는지 직관이 없음.
  - 결혼 스타일 무관하게 모든 가이드 노출 → 셀프웨딩 사용자에게 스드메 가이드.
  - 완료해도 다음에 무엇을 해야 할지 가이드 없음.
- **개선 (이번 PR)**:
  1. 챕터 구조 도입 (`tutorialChapters.ts`) — 기초 / AI / 예산·일정 / 커뮤니티·커플 / 프리미엄.
  2. 스타일 인지 필터 — 각 챕터/레슨에 `requiresStyles?` 옵션. 셀프웨딩에서 스드메 챕터 자동 제외.
  3. 진행률 추적 — `useTutorialProgress`가 localStorage에 레슨 완료를 저장.
  4. 새 튜토리얼 페이지 — 진행률 링 + 챕터 카드 + "이어서 시작" 버튼.
  5. 첫 방문 시 자동 노출되는 환영 시트(`TutorialWelcomeSheet`) — 결혼 정보 입력 완료 직후 1회.
  6. 깨진 셀렉터 수정 — 잘못된 `nav-ai` 단계 제거, AIPlanner에 `data-tutorial` attr 추가.
  7. PersonaDashboard에 튜토리얼 진행률 노출 — 미완 챕터 1개를 우측 상단 작은 칩으로.

### 3.2 AI 플래너 (백로그)
- 페인: 첫 질문 전송률 낮음 / 같은 질문 반복 시 컨텍스트 손실.
- 개선: 최근 질문 기록 표시, 컨버세이션 저장 옵션.

### 3.3 AI 스튜디오 (백로그)
- 페인: B1·C1·C2가 핵심인데 결과 공유·저장 동선 길음.
- 개선: 결과 → 커뮤니티 글쓰기 1탭 공유.

### 3.4 스케줄 / 예산 (직전 PR에서 이미 스타일 제외 처리됨)
- 페인: 새 일정 추가 UX가 무겁다.
- 개선: 빠른 메모 위주 카드 추가.

### 3.5 커뮤니티 (백로그)
- 페인: 글쓰기 진입 후 사진 업로드 동선이 분산.

### 3.6 프리미엄 PDF (백로그)

### 3.7 포인트·하트 (백로그)

## 5. e2e 전체 시뮬레이션 운영 (월 1회 — 방법론 고도화)

> 정식 운영: `.github/workflows/monthly-e2e-simulation.yml` 이 **매월 1일** Playwright e2e +
> 개발 에이전트(Claude) 시뮬레이션을 돌린다. 결과는 `docs/<YYMMDD>_e2e_simulation.md`.
> (주간 전체 코드감사는 `weekly-audit.yml`.) 둘 다 `ANTHROPIC_API_KEY` 시크릿 필요.

### 5.0 원칙 — "경쟁사 분석 선행"
새 기능/불편 영역을 시뮬레이션하기 **전에** 같은 기능을 서비스하는 **타 업체를 먼저 분석**한다.
우리만의 기준으로 보면 "동작하니 OK"가 되지만(SusVibes), 시장 표준과 대조해야 **구조적 마찰**이
드러난다. 분석은 별도 문서로 남기고 시뮬레이션 체크포인트의 근거로 삼는다.
- 예: 청첩장 편집 → `docs/260614_invitation_editor_competitor_analysis.md`.
  **핵심 교훈**: 한국 모바일청첩장 표준은 "폼 입력 + 고정 템플릿 실시간 미리보기"인데 Dewy 는
  자유 캔버스(Konva) — 시장 역방향이라 "편집이 불편"의 구조적 원인. 모바일 경로는 폼/섹션 기반을
  기본으로, 자유 캔버스는 종이(300dpi 인쇄) 경로로 분리하는 방향이 권고됨.

### 5.1 시뮬레이션 절차 (현재 코드 기준 · stale 스냅샷 불신)
1. 핵심 여정을 **페르소나별**로 surface walkthrough(가입·온보딩·홈 진입 팝업·청첩장 편집/발행·
   예약 문의·결제·커뮤니티). 각 단계 ✅(매끄럼)/⚠️(마찰)/❌(막힘) 표기.
2. 경쟁사 분석 체크포인트와 1:1 대조(우리에 없는 표준 패턴 = 마찰 후보).
3. dead-end UI 차원 동시 점검(toast/안내만, no-op, "준비 중" 잔존 — AGENTS 검증 섹션).
4. 발견사항 → 우선순위(P0/P1/P2)와 "빠른 개선 vs 구조 개선" 구분. 저위험만 즉시 수정, 나머지 deferred.

### 5.2 청첩장 편집기 체크포인트 (경쟁사 분석 발)
- **진입**: 템플릿 선택 → 입력까지 클릭 수/이탈 지점. "10분 완성" 체감되나.
- **텍스트 편집**: 인라인 vs 캔버스 드래그. 키보드가 편집 대상을 가리지 않나(해외 대표 불만).
- **사진**: 업로드·교체·자르기 인터랙션, 터치 핸들 크기, 정렬 가이드.
- **터치 최적화**: 작은 슬롯을 손가락으로 정확히 집을 수 있나, 줌, 실행취소(undo) 접근성.
- **저장/공유/출력**: 발행→공유 링크, 종이는 300dpi/bleed/safe-margin(invitation-template-rules).
- **마법사 흐름**: 단계별 입력 → 실시간 미리보기가 시장 기대. 자유 캔버스 강제 시 마찰.

### 5.3 한계
브라우저/실기기 e2e 가 막힌 환경(샌드박스)에선 "검증함" 보고 금지 — 정적/코드 기준 분석임을 명시하고
실환경 확인을 요청한다(verification-lessons 규칙). 경쟁사도 1차 사용성 테스트가 막히면 후기·공식
문서 기반임을 명시.

# 듀이(Dewy) 회사·서비스 개요 (인수인계 / Handoff)

**작성일** : 2026. 06. 28. ｜ **목적** : 새 협업자·에이전트·투자검토자가 이 문서 하나로 "듀이가 무엇이고,
어디까지 왔으며, 어디를 보면 되는지"를 빠르게 파악하도록 하는 **온보딩 단일 진입점**.

> 본 문서는 요약·길잡이다. **정량 지표의 진실원천은 아래 원문 문서와 운영 DB·코드 저장소**이며,
> 수치는 별도 명기가 없는 한 창업도약패키지 사업계획서(`docs/260628_dochak_business_plan.md`,
> 2026. 06. 28. 운영 DB 직접 조회 기준)를 인용한다. 이 문서를 사실의 원천으로 삼지 말고,
> 판단이 필요하면 항상 원문·코드·DB 를 교차 확인할 것(AGENTS.md §검증 원칙).

  충남 천안 소재 여성·1인 창업기업.
- **왜 특별** : ① 20모드 페르소나 초개인화, ② AI 이미지 생성 6종(시장 공백), ③ 자체 설계
  L1~L5 다층 환각차단 RAG, ④ 업체 4,378개 데이터 자산 — 단기 모방이 어려운 기술·데이터 해자.

## 3. 제품 개요 — 무엇을 하는가

한 코드베이스에 **세 역할(소비자·기업·운영자)**이 모이고 **런타임 역할 게이트**로 갈린다.

- **소비자(Consumer)** : 결혼 준비 당사자 대상. 탐색·꿀팁·준비도구·AI(플래너·시뮬레이션)·
  청첩장·쇼핑/견적문의·커뮤니티. 20모드 페르소나로 화면·추천이 분기된다.
- **기업(Partners, `/business/*`)** : 웨딩 업체 대상 B2B 콘솔(가입→인증→상품등록→검수→정산).
  상세관리·갤러리·상품·리뷰·쿠폰·리드·문의·배송·디자인·가이드.
- **운영자(Console, `/admin/*`)** : 업체·모더레이션·청첩장에셋·AI 운영·마케팅/CS.

**핵심 기능군**
- **AI 시뮬레이션 6종**(운영) : 웨딩컨설팅·스드메 통합·드레스·메이크업·헤어·종이청첩장.
  9종 목표(모바일청첩장·촬영시안·식전영상 3종 설계 완료).
- **AI 플래너 / 웨딩컨설팅 리포트** : L1~L5 환각차단 RAG 기반 대화·리포트.
- **20모드 페르소나 초개인화** : 임신·국제결혼·재혼·스몰·셀프·1인 진행 등 상황·성향 분류
  (`src/lib/weddingPersona.ts` 단일 레지스트리, 결정적·멱등).
- **결제** : 웹 카카오페이 + 네이티브 인앱결제(애플·구글), 영수증 서버검증·멱등 처리.

## 4. 핵심 차별성 / 기술 자산

1. **L1~L5 다층 환각차단 RAG(자체 설계)** — ① 의도 분류 후 DB 직접응답(LLM 미호출)
   ② 지역·항목 평균가·실업체 데이터 근거 주입 ③ 금액 출처태그 강제 ④ 출력 후처리 감사
   ⑤ 메모리 검증. 신뢰성 중심 도메인(금융·의료 등)으로 이전 가능한 핵심 IP.
2. **20모드 페르소나 초개인화 엔진** — 단위테스트로 정합성 검증.
3. **멀티모달 이미지 생성 파이프라인(6종)** — 이용자 사진 + 참조 이미지 합성으로 결과물 사전 제시.
4. **머신러닝 분류 모델(KLUE-RoBERTa)** — 콘텐츠 토픽 자동 분류(학습 완료, 배포 예정).
5. **멀티프로바이더 추상화 + 평가 인프라** — 모델 교체 용이, 프롬프트 캐싱(입력비 ~50% 절감).
6. **보안·개인정보 = 설계 원칙** — RLS 350개(87개 마이그레이션)로 서버 인가, 보안 권고 전수
   점검(즉시 수정 0건), AI 처리 사진 자동 파기·탈퇴 즉시 파기. (인가는 클라 가드가 아니라 **RLS 가 최종 책임**.)

## 5. 실증 지표 (2026. 06. 28. 운영 DB 조회 기준 — 마케팅 미집행 상태)

> 출처 : `docs/260628_dochak_business_plan.md` Ⅲ-1. 아래는 **정식 출시 약 5주·광고비 0원**의
> 자연 유입 초기 실증값이다(핵심 가설 검증 목적, 규모 지표 아님).

| 구분 | 지표 |
|---|---|
| 제품 규모 | 화면 183개 / 라우트 135개 / DB 테이블 147개 / 서버리스 함수 60여 종(AI 약 20종) |
| 데이터 자산 | 등록 업체 **4,378개소**(11개 카테고리, 좌표 보유 3,949개소/90%), 콘텐츠 **16,131건** |
| 이용 지표 | 가입자 49명(최근 30일 46명), MAU 49·WAU 19 |
| 수익화 | 하트 거래 89건(유료 구매 28건), 포인트 거래 192건 — 유료 결제 정상 작동 검증 |

지원금은 "될지 모르는 가설"이 아니라 **검증된 전환 구조의 확대(마케팅·인력)**에 투입.

## 6. 수익모델

- **B2C** : 구독(첫 1개월 무료 후 정기, 마진 90%+) · 하트 토큰 5종(1,900~19,900원, 마진 67~97%) ·
  AI 외주(식전영상 등). 무료 이용자 일 5회 AI 질의 한도 → 유료 전환 유도.
- **B2B** : 업체 입점·노출·광고, 사업자 앱.
- **채널** : 웹 카카오페이 / 네이티브 스토어 인앱결제(웹 가격 +10%) 멀티채널.
- **확장** : 결혼 완료 후 신혼·기념일·육아 라이프스테이지 이동 → 활성기간 6→24개월, LTV 3~4배.

## 7. 코드베이스·아키텍처 (작업 전 필독)

- **스택** : Vite + React 18 + TS + Supabase + Capacitor(Android). 진입 `src/main.tsx`(react-router).
- **명령** : 빌드 `npm run build` · 테스트 `npm run test`(vitest) · 린트 `npm run lint`.
  Edge functions(`supabase/functions/*`)=Deno, 로컬 검증은 esbuild(AGENTS.md 참조).
- **구조** : 단일 레포·단일 빌드. 코드는 도메인 폴더로 분리(**Phase 1 완료**) —
  `src/features/{consumer,partners,console}/**` + shared(`src/lib`·`src/components`·`src/hooks`·
  `src/types`·`src/contexts`·`src/integrations`·`supabase`). **경계는 린트로 강제**(feature 간 직접 import 금지).
- **빌드 현실** : 웹·Android·iOS 는 **같은 `dist/` 번들**. Capacitor 가 래핑할 뿐 다른 코드 빌드 아님.
  Mac 은 별도 빌드 아니라 브라우저 웹 지원. 물리 분리(네이티브에서 console/partners 제외)는 로드맵 Phase 2.
- **백엔드** : Supabase 1개 공유(분리 금지). **인가는 RLS 가 끝까지 책임**(클라 가드는 UX용).
- **DB 진실원천** : `src/integrations/supabase/types.ts`(실 DB 생성 타입). DB 작업 전 테이블 선확인 게이트 필수.

## 10. 인수자 유의사항 (한계·주의)

- **수치의 유효기간** : 5장 지표는 2026. 06. 28. 스냅샷이다. 시간이 지나면 **반드시 최신 운영 DB
  로 재확인**하고 이 문서를 갱신할 것(이 문서를 캐시된 사실로 인용 금지).
- **미구현 항목 표기 유지** : 원문은 구현/부분구현/계획을 구분한다. 인수 시에도 "코드 완비 ≠ 운영
  가동"을 혼동하지 말 것(예: 인앱결제는 코드 완비·검증 단계, ML 분류는 학습 완료·배포 예정).
- **작업 경계 준수** : 한 영역(consumer/partners/console) 작업 시 다른 영역 파일을 함께 고치지
  않는다. shared 변경은 전 영역 파급 — grep 으로 사용처 확인 후 최소·표적 수정(AGENTS.md §경계 규칙).
- **"작동한다 ≠ 검증됨"** : 빌드·린트·단위테스트 통과만으로 완료 보고 금지. 사용자 시점 e2e 확인,
  불가 시 한계 명시(AGENTS.md §검증). placeholder CTA(죽은 토스트)·dead-end UI 는 별도로 점검.

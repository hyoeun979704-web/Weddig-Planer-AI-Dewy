> DEWY 의 모든 UX 변경(신규 기능·필드·단계·화면) 검토의 고정 기준점.
> 모든 UX 관련 PR 의 자기검문·코드리뷰가 본 문서를 따른다.
>
> v2 (2026-05) 갱신: 3대 원칙 명시화 + 정보 수집 위계 도입 + Google 의 정보 수집
> 방식(행동 신호·Just-in-time·가치 교환·점진적 옵트인) 일부 흡수 + 위치 정보

>
> v1 (페르소나 v1 모달 사례에서 도출) 의 6 질문·안티 패턴은 보강 유지.

## 0. 3대 원칙 + 메타 원칙

### 3대 원칙 (Priority Order)

UX 변경은 다음 세 가지 가치를 **이 순서대로** 추구한다:

1. **불편 해소 (Friction Reduction)** — 사용자가 "막혔다"고 느끼는 모든 지점 제거. 알 수 없는 에러·무음 실패·되돌릴 수 없는 동작·찾을 수 없는 기능 0건이 목표.
2. **결정 피로 감소 (Decision Fatigue Reduction)** — 사용자에게 요구하는 의사결정 횟수 최소화. 기본값·자동 분기·페르소나 매칭이 일차 수단.
3. **개인화 (Personalization)** — 같은 사람에게 같은 화면이 안 보이게. 사용·진행에 따라 누적된 정보로 매번 더 맞춤. 페르소나 시트 v1 P1~P20 분기 + 행동 기반 가중치 모두 활용.

세 원칙이 충돌하면 **위에서 아래 순으로** 우선. 예: "더 정확한 개인화" 가치를 위해 "한 번 더 입력 받는 불편" 을 부과하면 위반.

### 메타 원칙: 티 안 나게 (Invisible Collection)

위 3대 원칙을 달성하는 핵심 수단: **사용자가 정보를 제공하고 있다는 사실을 의식하지 않게**.

직접 입력 폼은 사용자의 의식·시간·결정을 모두 요구한다. 사용 흐름 안에서 자연스럽게 수집되는 신호는 그 셋 다 비용 0. **모든 정보 수집은 비용 순으로 위계를 따른다** (§1).

## 1. 정보 수집 위계 (Collection Hierarchy)

사용자에게 직접 묻는 것은 **마지막 수단**. 우선순위:

| 단계 | 비용 | 패턴 | DEWY 예시 |
|---|---|---|---|
| **L1 행동 신호** (Behavioral) | 0 | 페이지 조회·체류 시간·스크롤·클릭 패턴·검색어 기록 | 임산부 드레스 카테고리 조회 3회 → `pregnancy_interest=true` 후보 신호 |
| **L2 맥락 추론** (Contextual) | 0 | 시간대·디바이스·세션 분기 | 평일 23시 사용 → 통근/야간 사용자 (P1 김지현 패턴) |
| **L3 기존 데이터 유도** (Derive) | 0 | 이미 받은 신호로 다른 신호 계산 | `ceremony_type='dual_ceremony'` → 자동으로 international 페르소나, 영문 자료 활성 |
| **L4 행동 + 부드러운 확인** (Behavioral + Soft Confirm) | 낮음 | 추론한 신호를 1탭 확인 ("맞으신가요?") | 임산부 드레스 5회 조회 후 "임신 관련 콘텐츠 받으시겠어요?" 옵션 |
| **L5 Just-in-Time 입력** (JIT Input) | 중 | 기능 진입 직전 그 자리에서 인라인 요청 | 견적 PDF 클릭 시 "예상 하객 수" 그 화면에서 1줄 입력 |
| **L6 온보딩 직접 입력** (Upfront Input) | 높음 | 모달·폼에 명시 입력 | 결혼 예정일·지역 (대체 불가한 필수 신호만) |

**구현 우선순위**: L1 → L6 순으로 검토. 상위 단계로 수집 가능하면 하위 단계는 쓰지 않는다.

### L1 — 행동 신호로 채울 수 있는 것

- `wedding_style` 후보: 어느 카테고리(일반홀/하우스/공공시설) 페이지 조회 빈도
- `wedding_region` 후보: 검색·찜한 업체들의 city/district 다수결
- `planning_stage` 자동 진전: 견적·예약·결제·계약 완료 클릭 시 자동 stage up
- `budget` 후보: 찜·비교한 업체들의 min_price 분포
- `guest_count` 후보: 청첩장 발송·게스트 리스트 입력 흐름에서 누적
- `marital_history='remarriage'` 힌트: 재혼 커뮤니티 진입/검색
- `pregnant` 힌트: 임산부 드레스·신혼여행 안전 가이드 조회
- `country` 힌트: 디바이스 locale·timezone·IP region (단, IP 는 부정확하니 힌트만)

수집한 행동 신호는 **확정** 아니라 **후보**로 저장 (§5 민감 정보 처리 참조).

### L2 — 맥락 추론

- 세션 시간대 분포 → 사용 가능 시간 추정 (푸시 시간 자동 조정)
- 디바이스 (모바일/데스크탑) → 자세한 비교는 데스크탑, 빠른 조회는 모바일
- 통근 시간대 짧은 세션 → "통근 중 잠깐 보세요" 같은 카드 우선
- 주말 긴 세션 → "오늘 한 시간 정리" 같은 깊은 비교 도구 우선

### L3 — 기존 데이터 유도

- `ceremony_type='dual_ceremony'` → `persona_mode='international'` 자동
- `pregnant=true` + `wedding_date` → 본식 시점 차수 자동 계산
- `wedding_region` 시도 → 시군구 후보 자동 제시 (확정 X, suggestion)
- 찜한 업체 평균 가격 → 예산 후보 제시

### L4~L5 차이

L4 (Behavioral + Soft Confirm) 는 **추론한 신호에 대한 확인**. "이게 맞아요?" 한 줄.
L5 (JIT Input) 는 **추론할 신호가 없을 때 그 자리에서 받기**. "지금 입력하시면 PDF 생성됨" 같은 가치 교환.

L4 가 L5 보다 우선 — 추론할 데이터가 조금이라도 있으면 항상 추론 + 확인 형태로.

## 2. 6 Questions → 7 Questions (자기검문)

신규 필드·단계·화면·토글을 추가하기 직전에 7개 질문. 하나라도 **불확실**이면 형태를 다시 고민한다.

| # | 질문 | 통과 기준 |
|---|---|---|
| Q1 | **꼭 필요한가** | 없으면 99%+ 사용자에게 가치 손실 발생? 아니면 1% 페르소나만의 가치? |
| Q2 | **지금 물어야 하는가** | 첫 화면 vs. 기능 진입 시점 (JIT) 중 어느 쪽이 적절? |
| Q3 | **누구에게 보이는가** | 전체? 페르소나별 조건부? 기존 신호로 분기 가능? |
| Q4 | **단계가 늘어나는가** | 기존 화면/섹션과 통합 가능? 인접 단계 합치기 검토? |
| Q5 | **반복 인터랙션인가** | N개 입력에 N번 클릭? 멀티·시드·추론으로 줄일 수 있나? |
| Q6 | **편집 시 숨겨지는가** | accordion 접힘이라도 기존 입력값이 있으면 자동 펼침 보장? |
| **Q7** | **직접 입력 안 받고 수집 가능?** | L1~L4 (§1) 로 대체 가능? 직접 입력은 진짜 최후 수단인가? |

**답이 "아니다" 가 하나라도 있으면 형태를 바꾼다** — 위계 상승·노출 조건 추가·자동 추론 도입. 그래도 안 되면 기능 자체를 보류한다.

## 3. 노출 전략 결정 트리

신규 UI 요소의 노출 위치는 아래 트리로 결정한다.

```
신규 UI 요소 ─→ 행동 신호로 수집 가능? (§1 L1)
                ├─ YES ─→ 행동으로 수집, UI 추가 안 함 (또는 L4 확인 카드만)
                │
                └─ NO  ─→ 모든 사용자에게 가치 있나?
                          ├─ YES ─→ 매번 보여야 하나?
                          │         ├─ YES ─→ 상단 고정 노출
                          │         └─ NO  ─→ Just-in-time (기능 진입 시점)
                          │
                          └─ NO  ─→ 어떤 신호로 분기 가능?
                                    ├─ persona_mode  ─→ 페르소나 분기 자동 노출
                                    ├─ pregnant      ─→ 임신 컨텍스트 카드
                                    ├─ planning_stage─→ 진행 단계 분기
                                    ├─ 행동 누적      ─→ N회 진입 후 카드 노출
                                    └─ 없음           ─→ accordion + "해당하시면" 카피
```

### 노출량 상한

| 화면 종류 | 첫 노출 권장 최대 |
|---|---|
| 온보딩 모달 | 필수 4 + 자주 쓰는 1 = **5 섹션** |
| 상세 페이지 hero | **3 정보 chip** |
| 페르소나 대시보드 | D-Day + 다음 액션 3개 + 미션 3개 |
| 필터바 quick chip | **5 칩** |
| 비교표 (AI 응답) | **5 컬럼** 최대 |
| JIT 권한 요청 | 한 번에 **1 권한** (위치만, 알림만, 사진만 — 한꺼번에 묶지 않음) |

## 4. 자연스러운 수집 패턴 (Implicit Collection Playbook)

§1 의 위계를 실제 구현에서 적용하는 구체 패턴.

### 4.1 Just-in-Time (JIT) — 권한·입력 요청

기능을 사용하려는 그 시점에 그 자리에서 요청. 첫 화면·온보딩에 미리 안 받음.

| 신호 | JIT 시점 | 가치 교환 카피 |
|---|---|---|
| 위치 권한 | 베뉴 카테고리 첫 진입 | "근처 식장을 자동으로 보여드릴까요?" |
| 알림 권한 | 첫 일정 추가 후 | "D-Day 7일 전 알림을 받아보시겠어요?" |
| 사진 권한 | AI Studio 시작 클릭 | (이미 PhotoUploadConsent 적용 — 패턴 일치) |
| `guest_count` | 견적 PDF 생성 클릭 | "예상 하객 N명 정도 보시나요?" (한 줄 입력) |
| `partner_name` | 커플 초대 생성 시 | "파트너 이름을 알려주세요" (선택, 비워도 진행) |

**카피 규칙**: "허용하시면 X가 됩니다" 형태. 권한 자체가 아니라 **얻는 가치** 를 앞에 둠.

### 4.2 가치 교환 (Value Exchange) Framing

권한·입력 요청은 항상 사용자가 **즉시 얻는** 가치와 1:1로 묶는다.

- ❌ "위치 권한을 허용해주세요" — 가치 불명
- ❌ "맞춤 추천을 위해 더 정확한 정보가 필요해요" — 추상적
- ✅ "근처 식장을 자동으로 보여드릴까요?" — 즉시 얻을 결과 명시

### 4.3 행동 신호 누적 + 확인 카드 (Behavioral + Soft Confirm)

L1 행동 신호로 후보 페르소나/속성을 추론한 뒤, 일정 횟수 이상이면 부드러운 확인 카드 노출. **반박 가능**해야 함.

```
사용자가 임산부 드레스 5회 조회 →
PersonaDashboard 에 작은 카드:
"임신 중이신 분들을 위한 가이드가 있어요. 받으시겠어요?"
[받기] [지금은 괜찮아요]
```

- 받기 → `pregnant=true` 설정 + 임신 콘텐츠 활성
- 괜찮아요 → 후보 신호 폐기, 같은 카드 30일간 노출 안 함
- 무응답 → 7일 후 한 번 더, 그 후 폐기

### 4.4 자동 진전 (Auto-Progress)

사용자의 명시적 액션이 단계 진전을 의미하면 `planning_stage` 등을 자동 업데이트. 사용자에게 "어디까지 진행하셨나요?" 다시 묻지 않음.

| 액션 | 자동 진전 |
|---|---|
| 첫 견적 받기 | `just_started` → `researching` |
| 첫 업체 계약 (외부 입력) | `researching` → `contracting` |
| 청첩장 발송 일정 | `contracting` → `wrapping_up` |
| 결혼식 D-Day 지남 | `wrapping_up` → `after_wedding` (라이프스테이지 전환) |

### 4.5 후보 → 확정 분리 (Candidate vs Confirmed)

L1~L3 추론값은 `*_candidate` 컬럼/state 로 저장. 사용자가 직접 입력하거나 확인 카드에서 받기 누른 값만 정식 컬럼에 영속화. 추론값이 잘못됐을 때 사용자 신뢰가 무너지지 않게 차단.

```sql
-- 예시 스키마
ALTER TABLE user_wedding_settings
  ADD COLUMN wedding_region_candidate TEXT,        -- 행동 추론
  ADD COLUMN wedding_region_candidate_source TEXT; -- 'location' / 'searches' / 'favorites'
-- wedding_region 은 사용자 확정값만
```

### 4.6 옵트인 정보 활용 동의 (Personalization Opt-In)

세팅 화면에 **단일 토글** "행동 기반 개인화" 노출. OFF 면 L1~L4 신호 사용 안 함 (직접 입력만). 기본값은 ON 이되 첫 화면에 명시.

PIPA 준거: 마케팅 동의·민감정보 동의와 분리해 별도 동의 기록 (§5 참조).

## 5. 민감 정보 수집 (Sensitive Information)

DEWY 가 다루는 민감 정보:

- **건강**: 임신, 산후, 장애, 알레르기 (드레스/식음 옵션)
- **가족 관계**: 재혼, 부모 부재, 자녀, 입양, 사별
- **신앙·종교**: 종교 예식, 결혼 의식
- **국적·체류**: 외국인 배우자, 비자
- **재정**: 자기/양가 소득 분담 가능 범위
- **건강 외 사적 사건**: 이전 결혼, 파혼 경험

이런 정보는 직접 묻기 부담이 크다. 다음 규칙 의무:

### 5.1 절대 강제 안 함

- 첫 화면·온보딩에 민감 정보 필드 **없음**. 모달 통과에 필수 아님.
- 비워두면 그냥 표준 흐름으로 진행 — 강요 안 함.
- 임신 같은 시간 민감 정보도 §4.3 행동 + 확인 패턴으로 수집 권장.

### 5.2 추론 + 부드러운 확인 우선

§4.3 패턴. 절대 추론값만으로 콘텐츠 분기/노출 안 함. 사용자가 확인한 후에만.

### 5.3 동의 기록 분리

기존 `user_consents.sensitive_health_pregnancy_v1` 패턴 유지. 민감 정보 카테고리마다 별도 동의 키. 사용자가 마이페이지에서 카테고리별 철회 가능.

```sql
-- 새 카테고리 추가 시
INSERT INTO user_consents (user_id, consent_type, consent_version, agreed, ...)
VALUES (?, 'sensitive_family_remarriage_v1', 1, true, ...);
```

### 5.4 잊혀짐 (Forgettable)

- 사용자가 마이페이지에서 카테고리별 토글로 OFF 시:
  - 민감 신호 컬럼 NULL 화
  - 관련 콘텐츠/미션 노출 중단
  - 동의 기록은 보존 (PIPA 의무) 하되 `revoked_at` 마킹
- "해당 정보를 잊었어요. 다시 알려주시면 언제든 켤 수 있어요" 한 줄 confirm

### 5.5 카피 톤

| 안 됨 | 됨 |
|---|---|
| "재혼이신가요?" | "두 번째 결혼이신 분들을 위한 가이드도 준비돼있어요" |
| "임신 중이세요?" | "임신 중에 결혼 준비하시는 분들을 위한 일정·드레스 옵션이 있어요" |
| "부모님이 안 계세요?" | "혼자 진행하시는 분들을 위한 가이드도 있어요" |

객체화·진단형 어조 금지. 사용자 본인이 자기를 인식하도록 선택지를 노출.

### 5.6 노출 시점

민감 정보 노출 카드는 다음 조건 모두 만족 시:
- 가입 후 ≥ 3일 (즉시 노출 X)
- 행동 신호 ≥ 3건 누적
- 같은 카드 30일 내 노출 1회 제한

## 6. 위치 정보 전략 (Location Information, v2 신규)

위치 정보는 가장 유용한 신호 중 하나지만, 권한·프라이버시 부담이 가장 크다.
다음 규칙 의무:

### 6.1 사용 케이스 (Use Cases)

DEWY 가 위치 정보로 얻을 수 있는 가치:

1. **베뉴/스튜디오 근처 자동 추천** — 권한 1회 사용, 결과 즉시 가치
2. **신혼집 후보 지역 추론** — 자주 가는 위치 분석 (지속 권한 필요)
3. **식장 투어 일정 알림** — 일정 일자에 근처 도착 시 메모/체크리스트 노출
4. **현 거주지 → wedding_region 후보 제시** — 시도 단위 단발 활용

❌ **하지 않을 케이스**:
- 사용자 행적 분석 (어디 갔는지 추적)
- 광고 타겟팅용 위치 누적
- 위치 기반 광고 푸시

### 6.2 권한 요청 패턴

**Just-in-time + 가치 교환 + 정확도 단계** 3원칙:

```
사용자가 [베뉴] 카테고리 첫 진입 →
  카드 노출: "근처 식장을 자동으로 보여드릴까요?"
  [한 번만 보기] [항상 허용] [수동 입력]

  - 한 번만 보기 → 1회 위치 요청, 그 화면에서만 사용, 저장 안 함
  - 항상 허용 → 지속 권한 (배경에서는 안 씀), 매 진입 시 사용
  - 수동 입력 → 시도 드롭다운으로 직접 (기존 흐름)
```

### 6.3 정확도 단계 (Granularity)

| 단계 | 데이터 | 사용처 |
|---|---|---|
| **시도** (광역) | "서울특별시" | wedding_region 후보 |
| **시군구** (좁힘) | "마포구" | 시군구 큐레이션 |
| **근처 N km** (정밀) | lat/lng 반경 5km | 식장/스튜디오 근접 정렬 |

기본 단계: **시군구**. 사용자가 명시 옵트인 했을 때만 lat/lng 사용.

### 6.4 저장 정책

- 1회 사용 (한 번만 보기) → DB 저장 0. 메모리에서만.
- 지속 권한 → `user_wedding_settings.wedding_region_candidate` 같은 후보 컬럼에만. 사용자 확정 시 본 컬럼으로 승격.
- 정밀 좌표 → 저장 안 함. 매 요청 시 메모리 사용 후 폐기.

### 6.5 카피 예시

```
[배지] 위치 권한 (한 번만)
"마포구 식장을 우선 보여드릴게요"
[허용] [수동 입력]

— 1초 후 결과 표시. 사용자 데이터 저장 안 함.
```

### 6.6 마이페이지 토글

세팅 화면:
- "위치 정보 자동 사용" — OFF 기본, 사용자가 명시 ON
- ON 일 때 다음 토글 노출:
  - "베뉴 진입 시 근처 자동 추천"
  - "현 거주지를 wedding_region 후보로 제시"
  - "식장 투어 일정 도착 알림"

각각 독립 토글.

## 7. 인터랙션 비용 가이드 (Cost Patterns)

신규 입력을 받을 때 적용하는 구체 패턴:

### 7.1 멀티 입력
- ✅ Calendar `mode="multiple"` 또는 멀티 select
- ✅ 빠른 시드 버튼 ("4주 주말")
- ✅ "전체 지우기" 회복 동작
- ❌ open → 1개 선택 → close 반복

### 7.2 자동 추론
- 사용자에게 묻지 말고 기존 데이터에서 유도 (§1 L3)
- 예: `ceremony_type='small_real'` 이면 wedding_style 자동 small 보정 검토

### 7.3 점진적 공개 (Progressive Disclosure)
- 5섹션 초과 모달은 `<details>` accordion 으로 그룹화
- 그룹은 **사용 목적** 기준 ("정확도 보강" / "해당하시면 알려주세요")
- 기존 입력값이 있으면 `open` 자동 활성 — 편집 케이스 보장

### 7.4 단계 통합
- 인접 wizard 단계가 **각 1회 인터랙션** 으로 끝나면 한 화면 통합
- type 토글 → 옵션 인라인 펼침 (fade-in) 패턴 권장

### 7.5 기본값
- 99% 사용자 케이스의 값을 default 로 (country='KR', has_parents_*=true 등)
- 미설정/기본값 케이스 카피는 "선택 안 함" 보다 자연어로 ("대한민국")

### 7.6 가독성 한도 (Readability Cap)
- 한 화면 텍스트 총합 ≤ 200자 (모바일 한 viewport 기준)
- tooltip/안내문은 ≤ 50자 한 줄

## 8. 안티 패턴 8선 (Anti-Patterns)

코드 리뷰에서 발견하면 차단.

### A1. 스키마 부풀음 (Schema Bloat by Feature)
**증상**: DB 컬럼 추가했으니 UI 입력 필드도 추가.
**처방**: 컬럼 도입 PR 과 UI 노출 PR 분리. UI PR 은 §3 노출 전략 트리 거침.

### A2. Wizard Sprawl
**증상**: 5+ 단계 wizard, 1회 인터랙션 단계 다수.
**처방**: 인접 단계 통합 검토. 단계 ≥ 5 면 통합 가능성 의무 검토.

### A3. Form Essay
**증상**: 한 화면에 10+ 필드, 스크롤로 끝까지 안 보임.
**처방**: 필수만 노출 + 정확도 보강은 accordion.

### A4. Picker Fatigue
**증상**: N개 값 입력에 picker N회 open/close.
**처방**: 멀티 모드 + 빠른 시드 (§7.1).

### A5. Toggle Wall
**증상**: 체크박스 10+ 개, 대부분 사용자는 default.
**처방**: 99% default 값으로 자동 처리. 토글 ≥ 5 면 그룹화·접힘.

### A6. Edit-Time Invisibility
**증상**: accordion 접힘 기본이라 사용자 입력값 안 보임.
**처방**: 모든 접힘 섹션은 `open={!!hasUserData}` 자동 펼침.

### A7. Interrogation Mode (v2 신규)
**증상**: 첫 화면·온보딩에서 직접 입력으로 모든 정보 받으려 함. L1~L4 (§1) 검토 안 함.
**문제**: 사용자가 "취조당하는" 인상. 결정 피로 폭증, 완료율 저하.
**처방**: 신규 필드 추가 시 Q7 (§2) 통과 필수. 직접 입력은 마지막 수단.

### A8. Sensitive Cliff (v2 신규)
**증상**: 임신·재혼·부모 부재 같은 민감 정보를 첫 화면에 노출. 강제 입력은 아니더라도 즉시 노출 자체가 부담.
**문제**: 사용자가 어플 자체를 닫음. "여기서 이걸 물어봐?" 거부감.
**처방**: §5 민감 정보 규칙 의무 — 가입 후 3일 + 행동 신호 3건 누적 + 부드러운 카피.

## 10. 회고 사례 (Case Studies)

### 10.1 페르소나 v1 모달 (v1 도출 배경)

**무엇이 일어났나**: 페르소나 모드 시스템(P0 #1) 도입 시 `WeddingInfoSetupModal` 에
신규 필드 5개를 일괄 추가. 모달 첫 화면 노출 섹션이 7 → 13.

**왜 잘못됐나**:
- Q1 위반: 5개 중 3개는 < 20% 페르소나용
- Q3 위반: 페르소나 분기 가능했는데 일괄 노출
- Q7 (v2 신규) 위반: 행동 신호로 수집 가능했음 (예: 신랑 vs 신부 자주 보는 카테고리)
- A3 + A7 위반

**어떻게 고쳤나** (이전 PR):
- 첫 노출 5섹션으로 복귀
- 신규 5개를 두 `<details>` accordion 으로 그룹화
- 편집 케이스 자동 펼침

### 10.2 향후 회피해야 할 패턴

신규 P2 작업(영문 청첩장·신랑 통합 여정·호텔 비교) 구현 시:
- 영문 청첩장: 사용자 device locale 이 영어권이면 자동 영문 옵션 제시 (L2 맥락). 직접 묻지 않음.
- 신랑 통합 여정: `role='groom'` 또는 신랑 카테고리 조회 빈도로 자동 진입. 명시 입력 안 받음.
- 호텔 비교: 사용자가 호텔 베뉴 2+ 곳 찜하면 비교 카드 자동 노출 (L1).

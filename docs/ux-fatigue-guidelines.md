# UX 검토 규칙 — 피로도 중심

> 본 문서는 DEWY 의 UX 변경(신규 기능·필드·단계·화면 추가/수정)을 검토할 때
> 적용하는 고정 규칙이다. 모든 UX 관련 PR 의 자기검문·코드리뷰 기준점.
>
> 작성 배경: 페르소나 v1 모드 도입 시 온보딩 모달에 신규 5개 필드를 한 번에
> 추가해 첫 화면 노출 섹션이 7→13개로 부풀었던 사례. **데이터 모델 필요성이
> UI 첫 화면 노출 사유가 되지는 않는다**는 교훈을 형식화한다.

---

## 0. 원칙 (Principle)

**페르소나의 페인 포인트를 해결하려고 사용자에게 새 피로를 안기지 않는다.**

피로(fatigue)는 다음 형태로 누적된다:
- 시각적 노출량 (한 화면에 보이는 섹션 수·필드 수)
- 의사결정 횟수 (선택지·토글·드롭다운 클릭 수)
- 단계 전환 (화면 이동·진행 막대 칸 수)
- 반복 인터랙션 (같은 작업의 N회 반복)
- 카피 길이 (안내문·라벨·tooltip 의 총 글자수)

신규 가치를 추가할 때 위 5개 모두를 측정하고, 증가분을 정당화하지 못하면
구현 형태를 바꿔야 한다. 정당화 못하면 **구현하지 않는 것이 옳다**.

---

## 1. 변경 전 자기검문 (Six Questions)

신규 필드·단계·화면·토글을 추가하기 직전에 6개 질문을 자신에게 던진다.
하나라도 **불확실**이면 구현 형태를 다시 고민한다.

| # | 질문 | 통과 기준 |
|---|---|---|
| Q1 | **꼭 필요한가** | 없으면 99%+ 사용자에게 가치 손실이 발생하는가? 아니면 1% 페르소나만의 가치인가? |
| Q2 | **지금 물어야 하는가** | 첫 화면에서 받을 가치 vs. 사용자가 그 기능을 처음 만나는 시점에 인라인으로 받을 가치 중 어느 쪽이 큰가? |
| Q3 | **누구에게 보이는가** | 모든 사용자에게? 특정 페르소나에게만 조건부? `wedding_style` / `persona_mode` 등 기존 신호로 분기 가능한가? |
| Q4 | **단계가 늘어나는가** | 기존 화면/섹션과 한 화면에 합칠 수 있는가? 인접 단계 통합 검토 했는가? |
| Q5 | **반복 인터랙션인가** | N개 입력에 N번 클릭이 필요한가? 멀티 선택·빠른 시드·자동 추론으로 줄일 수 있는가? |
| Q6 | **편집 시 숨겨지는가** | accordion·접힘 기본이라면 기존 입력값이 있을 때 자동 펼침을 보장하는가? |

**답이 "아니다" 가 하나라도 있으면 형태를 바꾼다** — 노출 위치·노출 조건·인터랙션
형태를 재설계. 그래도 안 되면 기능 자체를 보류한다.

---

## 2. 노출 전략 결정 트리

신규 필드/카드/배너의 노출 위치는 아래 트리로 결정한다.

```
신규 UI 요소 ─→ 모든 사용자에게 가치 있나?
                ├─ YES ─→ 매번 보여야 하나?
                │         ├─ YES ─→ 상단 / 첫 화면 고정 노출
                │         └─ NO  ─→ 인라인 노출 (기능 진입 시점)
                │
                └─ NO  ─→ 어떤 신호로 분기 가능?
                          ├─ persona_mode  ─→ 페르소나 분기 (PersonaDashboard 등에 자동)
                          ├─ pregnant      ─→ 임신 컨텍스트 (PregnancyHoneymoonBanner 등)
                          ├─ 입력값 존재    ─→ 빈 상태 vs 채워진 상태 분기
                          ├─ 진행 단계      ─→ planning_stage 기반 분기
                          └─ 없음           ─→ accordion (접힘 기본) +
                                              "해당하시면" 카피로 무시 가능 표시
```

### 노출량 상한

| 화면 종류 | 첫 노출 권장 최대 |
|---|---|
| 온보딩 모달 | 필수 4 + 자주 쓰는 1 = **5 섹션** |
| 상세 페이지 hero | **3 정보 chip** (가격·평점·위치) |
| 페르소나 대시보드 | D-Day + 다음 액션 3개 + 미션 3개 |
| 필터바 quick chip | **5 칩** (region, 가격, 인원, 별점, 홀유형) |
| 비교표 (AI 응답) | **5 컬럼** 최대 (업체·별점·후기·시작가·위치) |

상한을 넘기면 그룹화 / accordion / "더보기" / 카테고리 분리 중 택1.

## 3. 인터랙션 비용 가이드

신규 입력을 받을 때 적용하는 구체 패턴:

### 3.1 멀티 입력
N개 일자/항목을 받을 때:
- ✅ Calendar `mode="multiple"` 또는 멀티 select
- ✅ 빠른 시드 버튼 ("4주 주말", "다음 3개월")
- ✅ "전체 지우기" 같은 회복 동작
- ❌ open → 1개 선택 → close → open → 다음 1개 선택 → close 반복

### 3.2 자동 추론
- 사용자에게 묻지 말고 기존 데이터에서 유도
- 예: `ceremony_type='small_real'` 이면 wedding_style 도 자동 small 로 보정 검토
- 예: 시도 선택 시 같은 시도의 시군구가 시드된 채라면 안 비움 (이전 사용자 의도 유지)

### 3.3 점진적 공개 (Progressive Disclosure)
- 5섹션 초과 모달은 `<details>` accordion 으로 그룹화
- 그룹은 **사용 목적** 기준 (예: "정확도 보강" / "해당하시면 알려주세요")
- 기존 입력값이 있으면 `open` 자동 활성 — 편집 케이스 보장

### 3.4 단계 통합
- 인접 wizard 단계가 **각 1회 인터랙션** 으로 끝나면 한 화면 통합 검토
- type 토글 → 옵션 인라인 펼침 (fade-in) 패턴 권장
- 단계가 줄면 진행 막대 자동 갱신

### 3.5 기본값
- 99% 사용자 케이스의 값을 default 로 (country='KR', has_parents_*=true 등)
- 사용자가 default 와 같으면 "선택하지 않은" 행동과 결과가 동일해야 함
- 미설정/기본값 케이스에서 노출 카피는 "선택 안 함" 보다는 자연어로 ("대한민국")

## 4. 안티 패턴 6선 (Anti-Patterns)

피해야 할 구체 패턴. 코드 리뷰에서 발견하면 차단.

### A1. 스키마 부풀음 (Schema Bloat by Feature)
**증상**: DB 컬럼 추가했으니 UI에도 입력 필드 추가.
**문제**: 데이터 모델 필요성과 UI 노출 사유는 별개. 데이터는 다른 채널(자동 추론, 후속 인라인 prompt) 로도 수집 가능.
**처방**: 컬럼 도입 PR 과 UI 노출 PR 을 분리. UI PR 에서는 노출 전략 트리(§2)를 거쳐 결정.

### A2. Wizard Sprawl
**증상**: 입력 받을 때마다 새 단계 추가, 5+ 단계 wizard.
**문제**: 각 전환에 페이지 이동·진행 막대·복귀 부담.
**처방**: 1회 인터랙션 단계는 인접 단계와 통합. 단계 ≥ 5 면 통합 가능성 의무 검토.

### A3. Form Essay
**증상**: 한 화면에 10+ 필드, 스크롤로 끝까지 안 보임.
**문제**: 첫인상 압도, 완료율 저하.
**처방**: 필수만 노출 + 정확도 보강은 accordion. "더 정확하게" 옵트인 패턴.

### A4. Picker Fatigue
**증상**: N개 값 입력에 picker N회 open/close.
**문제**: 의도 한 번에 표현 못 함, 손가락 피로.
**처방**: 멀티 모드 + 빠른 시드 (§3.1).

### A5. Toggle Wall
**증상**: 체크박스 10+ 개를 한 화면에 나열, 대부분 사용자는 모두 default.
**문제**: 사용자가 "이걸 다 봐야 하나" 부담.
**처방**: 99% default 값으로 자동 처리. 토글 카운트 ≥ 5 면 그룹화·접힘.

### A6. Edit-Time Invisibility
**증상**: accordion 접힘 기본이라 사용자가 이전에 입력한 값이 안 보임.
**문제**: 편집 케이스에서 "내 데이터 사라졌나" 인상. 실수 수정 불가.
**처방**: 모든 접힘 섹션은 `open={!!hasUserData}` 로 자동 펼침. 변경 사항 표시.

## 5. PR 자기보고 항목 (Self-Report)

UX 변경 PR 의 description 에 아래 항목을 명시한다. 코드 리뷰가 이걸 검증.

```markdown
## 피로도 영향

- **추가 인터랙션**: [클릭 N → M, 입력 N → M, 화면 전환 N → M]
- **첫 화면 노출 섹션 수**: [변경 후 N개]
- **노출 조건**: [모든 사용자 / persona_mode=X / pregnant=true / 입력 있을 때만]
- **단계 통합 검토**: [통합했음 (어떻게) / 통합 불가 이유]
- **자동 추론 가능성**: [기존 신호 X 에서 유도 가능 / 직접 입력 필요한 이유]
- **편집 케이스 보장**: [기존 입력값 자동 펼침 적용 / 해당없음]

## 회피한 안티 패턴 (해당 항목 체크)
- [ ] A1 스키마 부풀음 — DB 컬럼과 UI 노출 분리 검토함
- [ ] A2 Wizard Sprawl — 인접 단계 통합 검토함
- [ ] A3 Form Essay — accordion / progressive disclosure 적용함
- [ ] A4 Picker Fatigue — 멀티/빠른 시드 적용함
- [ ] A5 Toggle Wall — 기본값으로 99% 처리함
- [ ] A6 Edit-Time Invisibility — 자동 펼침 보장함
```

해당 안 되는 항목은 명시적으로 "N/A — 사유" 작성.

## 6. 회고: 페르소나 v1 모달 사례

본 가이드라인이 도출된 실제 사례.

### 무엇이 일어났나
페르소나 모드 시스템(P0 #1) 도입 시 `WeddingInfoSetupModal` 에 신규 필드 5개
(시군구·역할·거주국가·부모 부재·식 형태) 를 일괄 추가. 모달 첫 화면 노출
섹션이 7 → 13 으로 약 2배 증가.

### 왜 잘못됐나
- **Q1 위반**: 5개 중 3개(역할·부모 부재·국제)는 비표준 페르소나 < 20% 만의 가치인데 100% 사용자에게 노출.
- **Q3 위반**: 페르소나별 분기 가능했는데도 일괄 노출.
- **Q4 위반**: 인접 섹션 그룹화·통합 검토 안 함.
- **A3 안티패턴**: Form Essay — 13 섹션 일렬.

### 어떻게 고쳤나
직전 PR `ux(persona): 핵심 플로우 피로 감소` 에서:
- 첫 노출 5섹션으로 복귀
- 신규 5개 + 기존 차수/임신 을 `<details>` 두 그룹으로:
  - "더 정확한 큐레이션 받기" — 시군구/역할/식 형태
  - "해당하시면 알려주세요" — 차수/임신/국가/부모 부재
- 기존 입력값 있으면 자동 펼침 (A6 회피)

### 이번 가이드라인이 막을 다음 사고
신규 P2~P3 우선순위 작업(영문 청첩장, 신랑 통합 여정, 호텔 비교 매트릭스 등)
구현 시 같은 패턴 반복 차단.

## 7. 살아있는 문서 (Living Document)

본 가이드라인은 새 사고가 발생할 때마다 갱신한다. 다음 트리거가 발생하면 본 문서를 업데이트:

- 코드 리뷰에서 새 안티 패턴 발견
- 사용자 인터뷰/베타에서 피로도 신호 (이탈·문의·"너무 많아요" 피드백)
- 분기 보정 시 페르소나별 완료율 < 60%

리뷰 주기: 매 분기.

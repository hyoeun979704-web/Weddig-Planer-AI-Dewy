# 코드 작성·리뷰 규칙 (바이브코딩 안전장치)

> 출처: "AI 기반 바이브 코딩 환경에서의 코드 안정성·완성도 검토 체계" 분석.
> CLAUDE.md 의 "## 코드 작성·리뷰 규칙"에서 요약·참조하는 **상세 기준**.

## 왜 필요한가 (배경)

AI 가 생성한 코드는 **기능은 통과해도 보안·안정성은 침묵 속에 깨진다.**

- SusVibes 벤치마크(CMU, 2025): 기능 통과율 80~91% vs 보안 통과율 8~23% — **중앙값 격차 45%p**, LLM 생성 코드의 **80% 이상이 새로운 보안 취약점을 도입**.
- 근본 원인: 컴파일러/테스트는 구문·타입 오류만 즉시 피드백한다. **SQL 인젝션·XSS·경로 탐색·권한 우회·정보 노출은 "조용히(silent)" 작동**하므로 모델이 "틀렸다"는 신호를 받지 못한다 → 통계적으로 자주 본 (불안전한) 패턴을 그대로 재현한다.
- 그래서 **"기능이 된다 ≠ 완성"**. 사람(=리뷰어)이 아래 6차원으로 적대적으로 심문해야 한다. 이 프로젝트의 CLAUDE.md "검증" 문화(실제 반영 확인 / DB 스키마 정합성)와 한 묶음이다.

규칙의 톤: **"~하면 좋다(should)"가 아니라 "~해야 한다(must)".** 애매하면 멈추고 확인한다.

## 6차원 검증 프레임워크 (우선순위 순)

코드를 새로 짜거나 바꿀 때, 그리고 리뷰할 때 **이 순서대로** 자문한다. 각 차원의 Red Flag 가 보이면 **고치기 전엔 "완료"라고 보고하지 않는다.**

### 1. 논리적 정확성·견고성 (Correctness) — 최우선

검증 질문:
- 기획된 비즈니스 요구사항·의도를 **기능적으로 완전히** 충족하는가?
- **엣지 케이스**(빈 배열, `null`/`undefined`, 0/음수/초대형 값, 경계값)에서 전체를 다운시키지 않고 **우아하게 실패(graceful degradation)** 하는가?
- 동시성(concurrency)에서 **경쟁 상태(race condition)·교착(deadlock)** 을 유발하지 않는가? (예: 클라 카운터로만 한도 체크 → 서버 원자적 증가 필요)
- 복수 테이블 변경 시 **트랜잭션이 원자적**이라 부분 쓰기(partial write)가 없는가? (실패 시 롤백/환불 대칭)

Red Flags:
- **에러를 삼키는 빈 `catch {}` / `catch (e) {}`** — 특히 결제·인증·데이터 쓰기 경로. 최소한 로깅 + 상위 전파 또는 명시적 복구.
- 배열 길이·존재 확인 없이 인덱스 접근(`arr[0]`, `data[0].x`).
- 비동기 흐름에서 **`await` 누락** → 처리 순서 꼬임/미완료 상태 읽기.
- 멀티탭/멀티요청에서 stale 캐시·중복 발급(idempotency 부재).

이 코드베이스 적용:
- 라우트/렌더 실패는 ErrorBoundary 로 잡아 화이트스크린 금지(`src/components/ErrorBoundary.tsx`).
- Supabase 쓰기는 실패 경로(환불·롤백)를 earn/spend 대칭으로.

### 2. 선제적 보안성 (Security)

검증 질문:
- 모든 **외부 입력**이 비즈니스 로직에 닿기 전에 검증·살균(sanitization)되는가?
- DB 접근이 **매개변수화 쿼리**인가? (문자열 병합 SQL 금지. PostgREST `.or()`/LIKE 는 escape 헬퍼 사용 — `src/lib/postgrestEscape.ts`)
- **인증(authentication)을 넘어 인가(authorization)** 를 확인하는가? "로그인했나?"가 아니라 **"이 사용자가 이 리소스/액션에 권한이 있나?"**. (예: 클라가 보낸 `user_id` 를 JWT `sub` 와 대조; service_role 전용 함수는 role 클레임 검증)
- API 키·토큰·시크릿이 **소스에 하드코딩**돼 있지 않은가? (`Deno.env`/`import.meta.env` 로만. anon/publishable 키는 예외)

Red Flags:
- 문자열 병합 SQL, 사용자 입력으로 만든 파일 경로(디렉토리 우회).
- **비밀번호/PII/요청 페이로드를 로그에 기록**, 에러 응답에 내부 스택/DB 스키마(컬럼명) 노출 → 클라엔 제네릭 메시지, 상세는 서버 로그만.
- `eval()`/`Function()`/동적 Base64 실행, 출처 불명·환각된 패키지 의존성.
- 와일드카드 CORS + credentials 동시 사용(토큰 인증이면 credentials 끄기).

이 코드베이스 적용:
- 클라이언트는 RLS 로 보호되는 게 전제 — UI 의 role/premium 게이팅은 **반드시 서버 RLS/SECURITY DEFINER RPC 로 뒷받침**. `user_roles` 등 권한 테이블은 클라 INSERT/UPDATE 정책 없어야(자가 권한상승 차단).
- SSR/HTML 주입은 `esc()` 로 이스케이프, `<script>`/JSON-LD 는 `<` → `<`.

### 3. 리소스·성능 최적화 (Performance)

검증 질문:
- 반복문 안에서 DB 를 매번 쿼리하는 **N+1** 이 아닌 단일 벌크/조인인가?
- 비싼 집계·외부 API 결과에 **캐싱**(React Query staleTime 등)이 있는가?
- 대규모 리스트는 **페이지네이션**(무한스크롤/limit)이 있는가? 큰 페이로드에 OOM 방지가 있는가?

Red Flags:
- 중첩 루프 O(n²)+, 인덱스 없는 전체 테이블 스캔.
- 프론트의 불필요한 state 변경·**렌더 폭주**(매 렌더 새 객체/새 Date 를 `useEffect` deps 에 → 무한 루프; `useMemo`/effect 내부 계산으로).

### 4. 테스트 커버리지·신뢰성 (Testing)

검증 질문:
- 새 로직의 **분기·실패 시나리오**를 입증하는 단위/통합 테스트가 있는가?
- 기존 CI 회귀 테스트(`npm run test` 276+)를 모두 통과하는가?
- **실제 운영 데이터와 유사한 동적 목업**으로 검증하는가? (`id=1` 같은 하드코딩 값에만 의존해 항상 성공하는 테스트 금지)

Red Flags:
- 통과만을 위해 조작된 테스트, 내부 구현에 강결합돼 사소한 변경에 깨지는 fragile 테스트(외부 인터페이스 기준으로 작성).

### 5. 가독성·유지보수성 (Maintainability)

검증 질문:
- 처음 보는 동료가 모듈의 의도·흐름을 쉽게 파악하는가?
- 명명 규칙이 **기존 컨벤션**을 따르는가? 복잡한 로직이 단일 책임(SRP)으로 작은 함수로 분할됐는가?
- **"어떻게"가 아니라 "왜"** 이렇게 짰는지(비자명한 결정의 배경)가 주석으로 남았는가?

Red Flags:
- 의미 추론 불가한 단일문자 변수 남발.
- **중복 로직 반복(DRY 위배)** — 같은 매핑/포맷/라벨이 여러 파일에 복붙되면 드리프트 버그(같은 값이 화면마다 다르게)로 직결. 단일 소스로 추출(`src/lib/*`).
- 명명 상수로 빼야 할 **매직 넘버** 직접 사용.

### 6. 시스템 아키텍처·API/인터페이스 설계

검증 질문:
- 외부 노출 API/인터페이스의 **매개변수 순서·데이터 구조·반환 타입**이 기존 생태계와 일관적인가?
- 계층형 패턴(Repository/Service/Controller, 이 프로젝트는 hooks/lib/components)을 침범하지 않고 **적절한 계층에 로직이 위치**하는가?
- 파괴적 변경(breaking change)이면 버저닝/피처 플래그 등 **점진적 이관 전략**이 있는가?

Red Flags:
- 도메인·DB 접근 로직이 UI 컨트롤러/컴포넌트에 뒤섞인 스파게티.
- 모듈 간 강결합(tight coupling)으로 분리 불가.

## 작성 시 기본 원칙 (스스로 지킬 것)

1. **페르소나 고정**: 코드를 짤 땐 "유지보수성·SOLID·보안에 집착하는 시니어 + 적대적 입력을 가정하는 DevSecOps" 시점으로 본다.
2. **맥락 우선**: 어디서 실행되고(브라우저/Edge/Deno), 기존 패턴이 무엇인지 확인 후 짠다. 엉뚱한 리팩토링·불필요한 라이브러리 도입 금지.
3. **단호한 자기 지시 + 부정 제약**: 요구 범위를 넘어 함수 전체를 다시 쓰지 말 것, 호출부 시그니처를 함부로 바꾸지 말 것. 변경은 **최소·표적화**.
4. **퓨샷 일관성**: 같은 동작은 이미 있는 헬퍼/패턴을 재사용(재구현 금지). 추가 전 `src/lib`·`_shared` 를 먼저 검색.
5. **자기 검증(self-reflection)**: 짠 뒤 "시간복잡도/취약점/엣지케이스를 다시 한 번 적대적으로 점검". SQL row count·타입체크 통과만으로 "정상"이라 보고 금지(→ CLAUDE.md 검증 규칙).

## 변경 완료 전 필수 체크리스트

코드 변경을 "완료"로 보고하기 전에 모두 통과해야 한다:

- [ ] 1. **Correctness**: 엣지케이스(빈/null/경계)·에러 경로 처리, 빈 `catch{}` 없음, `await` 누락 없음.
- [ ] 2. **Security**: 입력 검증, 매개변수화 쿼리, **인가** 확인, 하드코딩 시크릿 없음, 로그/에러에 PII·내부정보 누출 없음.
- [ ] 3. **Performance**: N+1 없음, 필요한 캐싱/페이지네이션, 렌더 폭주 없음.
- [ ] 4. **Testing**: 분기/실패 테스트, `npm run test` 통과, 현실적 목업.
- [ ] 5. **Maintainability**: 컨벤션 준수, DRY(중복 로직 단일화), 매직넘버 상수화, "왜" 주석.
- [ ] 6. **Architecture**: 계층 준수, API 일관성, breaking change 시 이관 전략.
- [ ] 7. **실제 반영 검증**(CLAUDE.md): end-to-end 동작/DB 스키마 정합성을 실제로 확인. 못 하면 한계를 솔직히 명시.

검증 환경 한계로 e2e 확인 불가 시 "검증함"이라 보고하지 말고 한계를 명시한다.

## (참고) AI 리뷰 프롬프트 템플릿

큰 변경/PR 은 아래 페르소나로 2차 자동 리뷰를 거는 것을 권장한다. 영어 프롬프트가 성능이 가장 좋다.

- **템플릿 1 — 아키텍처 완결성** (Staff Engineer): SOLID/DRY/KISS, 숨은 기술부채·아키텍처 시한폭탄. 출력=Markdown(Executive Summary / Critical Blockers / Optimization / 최소 diff). 제약: 전체 재작성 금지, 사소한 스타일 지적 금지.
- **템플릿 2 — 방어적 보안/공급망** (DevSecOps): 인젝션(CWE-79/89) 데이터 흐름 추적, 인가/시크릿, 정보노출(CWE-200/532), 위험 호출(eval)·타이포스쿼팅. 출력=CI 통합용 JSON(`security_posture: SAFE|SUSPICIOUS|CRITICALLY_VULNERABLE`, `pipeline_action: BLOCK_MERGE|APPROVE`).
- **템플릿 3 — 파괴적 테스트** (SDET): 암묵적 가정 매핑, 경계/음수/유니코드/타임아웃 등 적대적 케이스, 폴트 톨러런스(롤백). 출력=Given/When/Then BDD 시나리오 3~5개.

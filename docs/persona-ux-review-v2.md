# DEWY 페르소나 v1 기준 UX 전면 검토 (v2)

대상 페르소나: 페르소나 시트 v1의 P1~P20 (표준 10인 / 스몰 4인 / 셀프·노웨딩 2인 / 스냅 1인 / 특수 3인)

대상 surface: 온보딩 · 홈/탭 · 검색 · 알림 · 찜 · 장바구니/결제 · AI 플래너 · AI 스튜디오 · 꿀팁 · 이벤트/혜택 · 쇼핑 · 스케줄 · 예산 · 커뮤니티 · 청첩장 · 커플 도구 · 마이페이지 · 프리미엄/구독

방법: 각 surface의 실제 코드(흐름·DB 스키마·게이팅·시스템 프롬프트)를 읽어 페르소나 페인 포인트 5개 × 20인 = 100개 항목에 대해 "현재 코드가 해결하는가"를 검증했다. 추측이 아니라 코드에 존재하는 동작·필드만 근거로 한다.

자기 인지: 페르소나 시트 자체가 가설이라 모든 매칭은 사용자 데이터 확보 시 재보정 대상이다. 본 문서는 "지금 이 코드가 페르소나의 페인 포인트를 어디까지 해결하는가"의 스냅샷이다.

기존 `docs/persona-ux-review.md`는 다른 페르소나 세트(P1 알뜰 지민 ~ P20 준호) 기준이라 본 문서로 대체하지 않고 병행 참조한다. 두 문서가 중복 지적하는 항목은 우선순위가 더 높다고 본다.

## 0. TL;DR — 가장 큰 갭 3가지

1. **페르소나 모드 시스템이 사실상 3-way에 머문다.** 온보딩에서 `wedding_style`을 `general/small/self/custom` 네 값 중에서 받지만 실질 분기는 미션·체크리스트 필터(`src/lib/weddingStyle.ts:36-57`)에만 작동한다. `marital_history(first/remarriage)`와 `pregnant`는 DB에 저장되고 AI 프롬프트(`supabase/functions/ai-planner/prompt.ts:131-175`)에만 반영될 뿐, UI/큐레이션/커뮤니티/스케줄·예산 도구에서는 분기가 없다. 시트의 P7(신랑), P9(해외), P10(부모 부재), P12(야외), P17(스냅), P20(국제) 6개 모드는 코드 어디에도 존재하지 않는다.
2. **지역 시스템이 17개 광역시도에서 멈춰 있다.** `src/data/regions.ts:13-30`에 광역시도만 등록되어 시군구 단위 큐레이션이 불가하다. P6(천안)·P12(양평) 같은 권역 통합 사용자에게 "충남" 결과가 통째로 노출되어 페인 포인트 그대로 재현. 검색 오버레이(`SearchOverlay.tsx:94-100`)는 이름 ILIKE만 지원해 "천안 식장" 같은 복합 쿼리도 막혀 있다.
3. **수집한 페르소나 신호의 절반이 미사용이다.** `pregnant`/`pregnancy_due_date`는 AI 프롬프트와 미션 일부만 분기하고, 스케줄 phase(`Schedule.tsx:439-523`)는 365~180일 고정 5단계라 P18의 "4개월 압축" 시나리오를 지원하지 않는다. `marital_history`는 DB에 박혀 있으나 (`src/components/wedding-planner/WeddingInfoSetupModal.tsx:412` 주석 "재혼 페르소나 톤 분기에 사용"이라고 명시) 실제 분기 코드를 찾지 못했다. 데이터는 받았는데 안 쓰는 상태.

이 3가지가 풀려야 시트의 비표준 페르소나(P11~P20, 1년차 추정 약 500명)에게 "내 결혼식 같지 않다"는 인상이 해소된다.

## 2. 횡단(Cross-cutting) 진단

페르소나별 진단으로 가기 전에, 여러 페르소나가 공통으로 부딪치는 시스템 단의 한계를 먼저 정리한다.

### 2.1 모드 시스템 (P0)

**현재**
- 입력 필드(`WeddingInfoSetupModal.tsx`)
  - `wedding_style: general | small | self | custom` (줄 65, 157-158)
  - `marital_history: first | remarriage | null` (줄 67, 159)
  - `pregnant: boolean` + `pregnancy_due_date` (줄 68-69, 160-162)
  - `excluded_categories: string[]` (줄 66, 158)
- 분기 코드
  - `src/lib/weddingStyle.ts:36-57`: small=한복/예복/대규모 청첩장 제외, self=스튜디오/메이크업/드레스 제외
  - `src/lib/personaMissions.ts:38-96`: 스타일별 미션 8~15개 분기
  - `supabase/functions/ai-planner/prompt.ts:131-175`: 재혼/임신 별 톤·맥락 시스템 프롬프트
  - `src/lib/pregnancy.ts` + `personaMissions.ts:98-158`: trimester별 미션

**부재**
- 신랑 모드(P7): `grep -ri "신랑모드\|groom_mode\|is_groom" src/` 검색 0건
- 국제결혼 모드(P20): `grep -ri "국제결혼\|international" src/` 검색 0건
- 부모 부재/원격(P9, P10): 0건
- 스냅 전용(P17): 0건 (스냅은 스튜디오 카테고리 일부)
- 야외 모드(P12): 0건 (태그 "야외웨딩"만)

**영향**: 시트 비중 50%인 표준 결혼식조차 신랑·부모부재는 모드 없음 → P7·P10은 첫 화면부터 "내가 아니다" 인상.

**제언**: `wedding_persona` 단일 enum (예: `standard_bride / standard_groom / standard_remote / small_real / small_outdoor / small_luxury / small_budget / self_no_ceremony / no_wedding / snap_only / pregnancy / remarriage / international / single_household`)을 받고, 각 surface에서 분기 훅(`usePersonaMode()`)을 통해 헤더 카피·큐레이션 가중치·자동 선택 카테고리를 분기. 기존 `wedding_style` + `marital_history` + `pregnant`는 정렬·필터·세부 분기에 그대로 사용.

### 2.2 지역 시스템 (P1)

**현재**
- `src/data/regions.ts:13-30`: 17개 시도만 (서울/경기/인천/…/제주)
- `normalizeRegion` ILIKE 매칭은 "충남"/"충청남도" 동치 처리 — 시도 수준에서만
- 검색(`SearchOverlay.tsx:94-100`): name ILIKE만, 지역+키워드 복합 ❌
- 예산 지역별 평균(`src/data/budgetData.ts:71-80`): 17개 시도별 평균 표시 ✅ — 큐레이션엔 미적용
- 허니문(`Honeymoon.tsx:19-22`): `initWithRegion(null)` — 사용자 거주지 필터 의도적 무효화

**영향**: P6(천안), P12(양평), P11(서울 종로 종로구 단위 레스토랑), P14(광주 구민회관), P9(싱가포르 거주) 모두 직격타. 페르소나 시트가 "지방 사용자 만족도 KPI"를 명시(P6 섹션)했는데 측정할 큐레이션 자체가 단층.

**제언**: 단기엔 시군구 컬럼 + 권역 그룹(수도권/충청권/호남권/영남권/제주) 추가, 검색 오버레이에 "{지역} {키워드}" 파서. 장기엔 허니문은 거주지 의존 X(맞음) → 대신 "출발 공항" 필드로 항공 시간/임산부 가능 여부 큐레이션.

### 2.3 큐레이션 신뢰 (P1)

**현재**
- 후기 인증 배지(`PlaceDetailLayout.tsx:656-660`): `is_verified=true` boolean → "인증" 마크 ✅
- 출처(`source_name`) 노출 ✅
- `place_reviews` 테이블 직접 조회(`usePlaceReviews.ts:22-40`) — 정적 하드코딩 아님

**부재**
- "광고" / "협찬" / "공동구매" 라벨 없음 — P3(가짜 후기 분간), P13(고급 후기 협찬 분리) 페인 그대로
- `places.recommended_for / pros / cons / hidden_costs`는 컬럼 존재(`PlaceDetailLayout.tsx:282-290, 520-542`)인데 입력 출처(업체 자작? 운영자 검증?)가 사용자에게 표시되지 않음

**영향**: P3·P13·P18 신뢰 즉시 훼손. P18(임신)은 "이 업체가 임산부 친화"라고 표시되어도 그게 업체 자작인지 후기 기반인지 모름.

## 7. 자기 인지·후속 조치

- 페르소나 분포 50/20/10/5/5/5/5%는 시트의 가설치다. 베타 5~10명 인터뷰로 실측 후 비표준 비중이 더 클 가능성도 있다(시트 §페르소나 분기 보정 계획).
- 본 문서의 ❌·⚠️ 판정은 "코드 검색·읽기"로 도달한 것이라, 운영에서 수동 큐레이터가 채우는 콘텐츠(예: places.recommended_for 입력)는 그대로 작동할 수 있다. 다만 표시 UX/필터/모드 분기는 코드 사실에 근거한다.
- 다음 분기에는 (1) `wedding_persona` enum 도입 후 4주 측정 (2) 시군구 분기 KPI (3) 임산부 드레스/일정 압축 KPI 세 가지를 베이스라인 vs 처치로 비교한다.

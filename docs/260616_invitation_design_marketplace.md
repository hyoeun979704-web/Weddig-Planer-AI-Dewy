# 청첩장 디자인 마켓플레이스 (작가 디자인 등록·판매) — 기획 + 정책/법무

> 제휴 청첩장 작가가 **디자인을 등록**하고 사용자가 **구매**해 에디터에서 쓰는 마켓플레이스.
> 기준 커밋 `6505ab2a`. **기획/검토 — 구현 전 결정사항(§6) 확정 필요.**
> (실물 인쇄 주문 설계 `260616_invitation_print_order_design.md` 와 짝: 디자인 구매 → 인쇄 주문 연결.)

## 0. 핵심 결론 (먼저)
- **디자인 구매 = 디지털 재화 → 인쇄 주문(실물)과 결제 정책이 정반대.**
  실물은 외부PG(카카오페이)·IAP 금지였지만, **디지털은 원래 Apple/Google IAP 영역**.
  **단, Dewy 는 이미 디지털(하트·프리미엄)을 카카오페이로 판매 중** → 같은 레일/정책 선례를 따른다.
  (한국 인앱결제법으로 제3자결제 허용 — §4. iOS 출시 시 컴플라이언스 재확인.)
- **기존 템플릿 시스템 재사용이 정답**: 작가 디자인 = `layout`(디자인데이터) 가진 템플릿 +
  소유자·가격·심사. 에디터는 거의 그대로 동작.
- 법적 핵심: **저작권(작가 소유·이용허락 범위)** + **작가 정산·원천징수** + **통신판매중개 고지**.

## 1. 사용자/작가 플로우
- **작가**: 디자인 등록(미리보기 이미지 + `layout` + 가격 + 스타일태그) → **운영자 심사** → 마켓 노출.
- **사용자**: 마켓에서 디자인 탐색(취향/스타일 필터·§미니퀴즈 연계) → 구매(결제) → **내 디자인**에
  추가 → 에디터에서 편집/발행 → (원하면) **인쇄 주문**으로 연결.
- **작가**: 판매 현황·정산 확인.

## 2. 아키텍처

### 재사용 (이미 있음)
- **템플릿 시스템**: `invitation_card_templates`·`mobile_invitation_templates`(`layout`·slug·two-sided·
  pricing rule). 에디터 `InvitationFlow`(template→wizard→result)가 템플릿을 로드. → 작가 디자인을
  **같은 형태(layout)** 로 저장하면 에디터 변경 최소.
- **디지털 결제 레일**: `kakao-pay-charge-*`(하트)·구독(프리미엄)이 이미 디지털 결제. 동일 패턴으로
  디자인 구매 처리(서버 금액·승인 시 권한부여). 하트/포인트 통화로 살지 직접 KRW 인지 선택(§6).
- 심사 패턴(콘텐츠 모더레이션)·업체 포털·알림.

### 신규 스키마
- `designer_designs` — 작가 등록 디자인:
  `id, designer_user_id, place_id?, title, description, preview_urls[], layout(jsonb 또는 template FK),
   price, style_tags[], status('pending'|'approved'|'rejected'), review_note, active, created_at`.
  (또는 기존 템플릿 테이블에 `owner_user_id·price·status` 컬럼 추가로 통합 — DRY 검토 §6.)
- `design_purchases` — 구매(이용권):
  `id, user_id, design_id, amount, order_id(결제 연결), licensed_at` UNIQUE(user_id, design_id).
  → 구매자는 해당 디자인을 에디터에서 사용 가능(라이선스 grant).
- `designer_payouts` — 정산:
  `designer_user_id, period, gross, fee, net, tax_withheld, status` (또는 거래별 ledger).

### 결제·권한 흐름
- `design-purchase-ready/approve`(또는 기존 charge 확장): **서버에서 가격 확정**(클라 금액 불신) →
  결제 → 승인 시 `design_purchases` insert(멱등·중복구매 방지) → 라이선스 grant.
- 무료 디자인은 결제 없이 즉시 grant. 이미 구매한 디자인 재구매 차단.

### 에디터 통합
- "내 디자인"(구매분) + 무료/기본 템플릿을 template 그리드에 함께 노출. 구매 안 한 유료 디자인은
  **미리보기 + 워터마크/잠금** → 구매 후 해제(미구매로 발행 차단 = dead-end 아닌 명확한 구매 CTA).

### RLS
- `designer_designs`: 공개 read(approved·active), 작가 본인 write(pending 로 시작), 운영자 심사.
- `design_purchases`: 본인 read; insert 는 결제승인 경로(definer)만(클라 임의 grant 금지 — 인가 핵심).

## 3. 화면
- 사용자: `/invitation/market`(디자인 그리드·필터) · 디자인 상세(미리보기·가격·구매) · "내 디자인".
- 작가: `/business/designs`(등록·상태·판매·정산) — 업체 포털 확장.
- 운영자: 디자인 심사 큐(admin).

## 4. 앱스토어 정책 (디지털 = 주의)
- **디지털 콘텐츠(앱 내 사용 디자인)는 원칙적으로 Apple IAP(3.1.1)·Google Play Billing 대상.**
  실물 인쇄와 정반대다.
- **현 상태**: Dewy 는 하트·프리미엄을 **카카오페이로 판매 중**(웹/PWA + 한국 인앱결제 우회 가정).
  디자인 마켓도 **같은 선례**를 따른다 → 신규 정책 리스크를 새로 만들지 않음.
- **한국 특례**: 전기통신사업법(2021) 개정으로 Apple/Google 이 **제3자 결제 허용**(국내). 단 수수료·
  컴플라이언스 존재. **iOS 정식 출시 시** 디지털 결제(하트·프리미엄·디자인) 전반을 함께 재점검.
- 권고: 디자인 결제는 **기존 디지털 결제 모듈과 동일 경로**로 통일(별도 IAP 도입 금지 — 일관성).

## 5. 법적 처리 (한국)
> **결정(260616)**: 라이선스 포함 가격은 **작가 본인이 책정** → 플랫폼은 법적 내용을 **고지(동의)**만
> 한다. 게시자(작가)는 **상품 등록 시 동의 게이트**(등록 버튼 → 약관 팝업 → 동의 체크 → 확인해야
> 등록), 소비자는 **구매 화면에서 이용범위·환불 고지**. 게이트 컴포넌트:
> `src/components/consent/DesignListingConsentDialog.tsx`(작가 등록용, 미체크 시 확인 비활성).
1. **저작권/이용허락** *(작가 책임·고지 방식)*:
   - 작가가 **저작권 보유**. 플랫폼은 **전시·판매 대행 라이선스**, 구매자는 **이용허락(라이선스)**
     — 범위 명시: "구매자 **본인 결혼식 청첩장 제작·인쇄 용도**, 재판매·재배포·2차 상업이용 금지".
   - 작가 **권리보증**(타인 저작권·폰트·사진 라이선스 적법) 서약 + **침해 시 책임·면책**.
   - **폰트/이미지 라이선스**: 디자인에 쓰인 폰트·스톡 이미지의 상업/임베드 라이선스 확인(흔한 분쟁원).
   - **통지·삭제(notice & takedown)** 절차 + 반복침해자 정책.
2. **통신판매중개**: Dewy 가 작가↔구매자 거래 **중개**면 "중개자·비당사자" 고지 + 작가(판매자)
   신원정보 제공(전자상거래법 §20). 정산 구조 결정.
3. **작가 정산·세무** *(중요)*:
   - 개인 작가에게 대금 지급 시 **원천징수**(사업소득 3.3% 또는 기타소득) + 지급명세서 제출.
   - 정산 주기·수수료(플랫폼 share)·세금계산서/현금영수증. 사업자/개인 구분 처리.
4. **청약철회(디지털)**: 디지털콘텐츠는 **다운로드·사용 개시 시 철회 제한** 가능(전자상거래법 §17②) —
   **사전 고지+동의** 필요. 미사용분 환불 정책 명시. (가입형/구매형 구분)
5. **표시의무·결제보호**: 가격·이용범위·환불조건 표시, 결제대금 보호(PG 구매안전).
6. **개인정보**: 작가↔구매자 사이 PII 노출 최소(정산 위해 작가에 필요한 최소정보만). 거래기록 5년.

## 7. 단계 (제안)
1. 스키마(designer_designs·design_purchases) + 작가 등록 UI + 운영자 심사.
2. 마켓 화면(그리드·필터·상세) + 미리보기 잠금.
3. 결제·라이선스 grant(기존 디지털 레일 확장, 서버 금액·멱등) + 에디터 통합.
4. 작가 정산·세무 처리 + 약관/라이선스/고지 화면(법무 §5).
> 결제·인가(라이선스 grant) 경로라 **정적 통과만으로 완료 보고 금지** — 실결제·권한부여 e2e 확인.

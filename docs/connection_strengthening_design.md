# 소비자↔제휴업체 연결 강화 — 설계안 (제안, 미구현)

> 목적: 소비자와 (제휴)업체를 잇는 경로를 강화. 현재는 "소비자가 업체를 찾아가 1:1 문의"하는
> **수동 경로**만 있다. 이를 **능동적 수요 라우팅 + 신뢰 신호 + 수요 기반 입점**으로 확장한다.
> 본 문서는 결정용 설계안이며 아직 구현하지 않음. 각 단계 effort·리스크·열린 질문 포함.

## 0. 현재 상태 (사실 기반)

**작동하는 1:1 경로**
- 소비자 → `PlaceInquirySheet`(인앱 문의) → `place_inquiries`(컬럼: status·answer·answered_at·
  contact·content·title·user_id·place_id) → `notify-inquiry`(엣지, 업체 알림) → `BusinessInquiries`
  (사장님 답변·상태).
- 보조: 파트너 랭킹(베프2>프렌즈1>일반0) 우선노출, 딜/쿠폰/이벤트, 후기 답변, 비즈 대시보드(조회·문의·후기).
- 입점 업체 문의 채널 선택(chat/url/phone), 미입점은 온라인채널→전화→안내 폴백(이번 세션 구축).

**핵심 공백**
1. **능동적 수요 라우팅 없음**: 1:N으로 수요를 뿌리는 경로(견적/매칭)가 없다.
2. **공급 병목**: 실데이터 claimed 업체 1곳, 활성 업체의 **78%가 연락처 없음** → 받을 업체가 부족.
3. **신뢰 신호 부족**: 상세에 응답률·응답속도·최근 문의수 없음 → "문의해도 답 올까?" 장벽.

매칭에 쓸 수 있는 기존 차원: `places.category`·`city`/`district`(지역)·`partner_rank`/`is_partner`,
사용자 `weddingPersona`(20모드)·`weddingStyle`·`wedding_region`·`wedding_date`·예산. (promotional_events
가 이미 `target_personas`/`target_styles` 타겟팅을 쓰므로 매칭 패턴 재사용 가능.)

## 1. (추천 #1) 견적 요청 / 매칭 — 1:N 수요 라우팅

**플로**: 소비자가 한 폼에 필요를 남긴다(카테고리·지역·예산·예식일·스타일·메모) → 조건 매칭되는
업체들에게 리드가 뿌려진다 → 업체가 견적/메시지로 응답 → 소비자가 상세 비교 후 선택.

**데이터 모델 (신규 2테이블, place_inquiries 패턴 답습)**
```
quote_requests        -- 소비자가 올린 견적 요청
  id, user_id, category, region_city, region_district,
  budget_min, budget_max, wedding_date, style, persona_mode, note,
  status('open'|'closed'|'expired'), created_at, expires_at
quote_responses       -- 업체의 응답(견적/메시지)
  id, request_id(fk), place_id(fk), owner_user_id,
  message, price_quote, status('sent'|'accepted'|'declined'),
  created_at, read_at
```
RLS: 요청자는 본인 요청+그에 달린 응답 read. 업체(owner)는 매칭된 요청을 read + 본인 응답 write.
매칭/팬아웃은 SECURITY DEFINER RPC(`create_quote_request`, `submit_quote_response`)로 — 직접쓰기 금지.

**매칭 로직 (RPC `match_partners_for_request`)**: 같은 category + 지역(city 우선, 없으면 광역) +
(style/persona 타겟이 있으면 일치). 정렬: partner_rank desc → 거리/평점. 상한 N(예: 20개 업체)으로
팬아웃 폭주 방지. 미입점 업체엔 리드 전달 불가 → **입점 유도 트리거**(§3과 연결).

**알림**: 매칭 업체에 `notify-inquiry` 재사용(리드 도착) + 응답 시 소비자에 알림(app_notifications).

**화면**
- 소비자: `/quote/new`(요청 폼, 카테고리 진입점·홈 CTA) · `/quote/:id`(받은 견적 비교 리스트).
- 업체: `BusinessLeads`(받은 리드 인박스 — 신규/응답함, 견적 작성).

**MVP 범위(축소)**: price_quote 없이 "상담 요청 → 매칭 업체에 알림 → 업체가 메시지 응답"만.
견적 금액·수락 플로는 Phase 2.

**Effort**: 중상(마이그레이션 2테이블+RPC 3, 소비자 2화면, 업체 1화면, 알림 연동). **리스크**: 공급
부족 시 "응답 0" 경험 → §2/§3 선행 또는 동시 필요. 스팸/남용은 요청 rate-limit(사용자당/일).

## 2. (추천 #3) 신뢰 신호 + 리드 파이프라인

문의 전환율을 올리고 업체가 리드를 "일하게" 만든다 — §1 없이도 독립 가치, §1과 시너지.

**소비자 측(상세 PlaceDetailLayout)**: 입점 업체에 **응답률·평균 응답시간·"빠른 응답" 배지·최근
문의 N건**(사회적 증거) 표시. 데이터는 place_inquiries 집계(answered_at-created_at).
→ "문의해도 답 온다" 신뢰 ↑ → 문의 시작률 ↑.

**업체 측(BusinessInquiries 확장)**: 문의/리드를 **상태 파이프라인**으로(신규→상담중→예약확정/종료) +
미응답 SLA 알림. place_inquiries.status 이미 존재 → 상태값 확장 + 칸반/필터 UI.

**Effort**: 중(집계 RPC/뷰 + 상세 배지 + 비즈 파이프라인 UI). **리스크**: 낮음(기존 컬럼 활용).
**선행 가치**: 데이터가 쌓일수록 좋아짐 — 먼저 깔아두면 §1 리드도 같은 파이프라인으로 흡수.

## 3. (추천 #2) 수요 가시화로 입점 유도 (공급 병목 해소)

연결의 전제는 "받을 업체"다. 이미 **조회·문의 시도가 발생하는 미입점 업체**의 잠재 수요를 입점
미끼로 전환한다.

- 미입점 상세/클레임 배너: "이번 달 **N명**이 이 업체를 봤어요 · **M명**이 문의하려 했어요 →
  무료 입점하고 직접 받으세요." (조회수 view_events + 문의 시도 이벤트 집계.)
- §1 매칭에서 미입점 업체가 매칭되면: 그 리드를 "대기 중인 문의"로 보관 → 사장님 입점 시 즉시 전달
  (입점 강력 동기).
- 효과: 우리가 *이미 가진 수요 데이터*(78% 미입점이라도 조회는 발생)를 공급 확보로 전환.

**Effort**: 소~중(집계 + 배너 카피 + 대기 리드 보관). **리스크**: 낮음. 단 "문의 시도" 이벤트
로깅이 필요(현재 미입점 문의는 외부 채널로 보내 우리 DB에 흔적 적음 → 클릭 이벤트 적재 추가).

## 권장 순서 & 페이징

- **Phase 1 (기반)**: §2 신뢰신호+리드 파이프라인 + §3 수요 가시화 입점 유도.
  → 공급(입점)·전환(신뢰)을 먼저 올려 §1의 "응답 0" 리스크를 줄인다. Effort 중, 리스크 낮음.
- **Phase 2 (핵심 레버)**: §1 견적 요청/매칭 MVP(상담요청 팬아웃 → 업체 메시지 응답).
- **Phase 3**: §1 견적 금액·수락·예약 전환 + ROI 대시보드(조회→문의→견적→예약 퍼널).

## 열린 질문 (결정 필요)
1. **§1 매칭 대상**: 입점 업체만 vs 미입점도(외부 채널로 리드 전달)? — 공급 부족이라 초기엔 둘 다 권장.
2. **견적 금액 노출 방식**: 공개 vs 1:1 비공개(역경매 우려). — 1:1 비공개 권장(업체 보호).
3. **스팸 방지**: 견적 요청 사용자당 일일 한도·인증 필요 여부.
4. **수익 모델 연계**: 리드/견적을 파트너 등급 혜택으로 묶을지(베프=리드 우선).
5. (릴리스) versionCode 6 Play 게시 여부 → 2.1.0 의 vc6 유지 vs vc7.

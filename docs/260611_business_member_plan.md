# 260611 기업회원(제휴업체) 시스템 — 현황 감사 · 경쟁 모델 분석 · 계획

## 1. 현황 감사 — "스키마가 없다"는 오해, 실제로는 거의 완성

DB(`information_schema`)와 코드를 실사한 결과, 기업회원 시스템은 **이미 구축돼
있고 미사용(전 테이블 0행) 상태**다:

| 구성요소 | 상태 |
|---|---|
| 스키마 | `business_profiles`(승인상태·심사메모·NTS검증·vendor 연결) + `business_events` + `business_coupons`(+`coupon_downloads`) + `business_products` + `inquiries` + `partner_deals` |
| 가입 | 회원가입 화면에 개인/기업 선택(`Auth.tsx` account_type) → `/business/onboard` |
| 검증 | `verify-business` edge fn — **국세청(NTS) 사업자번호 진위 확인** |
| 승인 | 운영자 어드민 `/admin/business-review` (승인/반려 + 사유) |
| 기업 포털 | `/business/*` 8페이지 — 대시보드(조회수·찜·미디어·쿠폰DL 통계), 업체정보(places 연동), 갤러리, 상품, 이벤트, 쿠폰, 문의함. `BusinessGuard` + 마이페이지 "업체 관리" 카드 |

→ **결론: 새 스키마 생성 불필요.** 진짜 문제는 "왜 0건인가" = 발견성·영업 채널과
운영 루프(채팅·알림·수익화)의 부재.

## 2. 경쟁 모델 조사 (3가지 수익·운영 모델)

**① 숨고 — 리드 과금 모델**: 가입 무료. 고수가 견적(리드)을 보낼 때 크레딧 차감
— 카테고리별 차등, 고객이 **열람해야 과금**(48h 미열람 시 보너스캐시 환급),
거래 성사 수수료는 0. 플랫폼 수익 = 리드 판매.

**② 당근비즈니스 — 무료 프로필 + 셀프서브 광고**: 비즈프로필 무료(가게정보·
소식·쿠폰·후기·**채팅**·단골 맺기) → 수익은 지역 타겟 광고(간편/전문가 모드,
AI 소재 생성). 동네 인증 기반 정밀 지역 타겟이 해자.

**③ 웨딩북Biz — 입점 SaaS** (docs/260611_competitor_analysis.md): 메인 플랫폼
프로모션 노출 + 고객관리·일정·통계 + 신부와 실시간 대화. "10만 고객·1,000
플래너" 홍보. 수익 = 입점/제휴 + 캐시백 연계 커미션.

**시사점**: 셋 다 ⓐ 무료 시작(진입장벽 0) ⓑ 업체 셀프서브 도구 ⓒ 수익화는
리드/광고/구독으로 분리. 그리고 셋 다 **실시간 채팅이 코어 루프**(문의→응답
속도가 전환율). Dewy 는 ⓐⓑ의 80%가 이미 있고 ⓒ와 채팅·알림이 없다.

## 3. 갭 분석 (중요도순)

| # | 갭 | 근거 |
|---|---|---|
| G1 | **입점 영업 채널 부재** — 가입 화면 안쪽에만 존재, 외부 업체가 알 길이 없음 (0행의 직접 원인) | 비로그인 입점 안내 랜딩/링크 없음 |
| G2 | **문의가 일방향 폼** — 채팅 부재, 새 문의 알림(푸시/이메일)도 없음 | inquiries 테이블만 존재 |
| G3 | **수익화 모델 미정** — 노출 부스트/리드 과금/구독 어느 것도 없음 | 과금 스키마·정책 부재 |
| G4 | 1업체 1계정 — 웨딩홀급은 직원 다계정 필요 | business_profiles.user_id 단일 |
| G5 | 통계 빈약 — 조회수 누계뿐, 기간별 추이·문의 전환 없음 | place view_count 스칼라 |

## 4. 계획 (단계별 — 큰 단위 PR)

**Phase B1 — 입점 진입로 (G1, 소)**
비로그인 입점 안내 랜딩 `/business`(혜택·절차·FAQ + 가입 CTA), 업체탐색
상세페이지에 "이 업체의 사장님이신가요?" 클레임 배너, 마이페이지 푸터에
입점 안내 링크. 제휴 영업 시 보낼 수 있는 단일 URL 확보가 목적.

**Phase B2 — 운영 루프 완성 (G2, 중)**
문의 스레드화(inquiries에 메시지 테이블 추가, 양방향 답글) + 새 문의/답글
이메일 알림(edge fn). 실시간 채팅은 기존 커뮤니티 인프라 검토 후 2단계.

**Phase B3 — 수익화 1차 (G3, 중)**
당근 모델 차용: 기본 무료 + "프로모션 노출" 유료(이벤트/혜택 탭 상단 고정,
기간제). 기존 promotional_events·하트 경제 재사용. 리드 과금(숨고형)은
문의량이 생긴 뒤 2차로.

**Phase B4 — 팀/통계 고도화 (G4·G5, 후순위)**
business_members(직원 초대) 테이블, 일별 노출·문의 통계 적재.

## 5. 진행 원칙
- 스키마 변경은 각 Phase 에서 필요한 시점에 멱등 마이그레이션 + MCP 적용 +
  롤백 트랜잭션 검증 (verification-lessons 준수).
- B1 은 코드만으로 가능(스키마 불요) — 첫 PR 후보.

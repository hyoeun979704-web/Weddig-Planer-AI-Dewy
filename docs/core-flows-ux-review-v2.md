## 1. F1 — 추가금 함정 사전 차단 (HiddenCostsCard)

### 현재 상태
- `src/components/detail/HiddenCostsCard.tsx`
- 트리거: `place.category ∈ {wedding_hall, studio, dress_shop, makeup_shop, honeymoon}` AND 상세 페이지 진입
- 노출: 업체별 hidden_costs + 카테고리 표준 체크리스트 4-6항목 + tip 한 줄

### v2 자기검문

| 항목 | 평가 |
|---|---|
| §1 수집 위계 | 정보 수집 0 — 패시브 읽기 콘텐츠. 위계 무관. |
| Q1 꼭 필요한가 | ✅ — P3·P13·P14·P18 핵심 페인. 그 외 페르소나도 가치 있음. |
| Q3 누구에게 | ⚠️ — 모든 사용자에게 매번 노출. P2 luxury hotel 페르소나는 N회 본 뒤엔 학습됨. |
| §5 민감 정보 | 해당 없음. |

### 안티 패턴 진단
- **A3 Form Essay 부분 위반**: 카테고리 표준 6항목 + tip = 약 100~150자. 모바일 viewport 텍스트 가독성 한도(§7.6 200자) 안에 듦. 단, hidden_costs 가 추가로 있으면 카드 1개가 300자 넘어갈 위험.

### 개선 제안

1. **첫 N회만 펼침, 이후 접힘** (§7.3 점진 공개 응용)
   - 같은 카테고리 상세 페이지를 3회 이상 조회한 사용자에겐 표준 체크리스트를 `<details>` 접힘으로 전환. "더 보기 / 항상 펼치기" 옵션.
   - localStorage 키 `dewy:hidden-costs-seen:{category}` 카운트.
2. **카테고리별 N개로 한정**
   - 현재 wedding_hall 6항목인데 가장 자주 발생하는 4개로 축약 가능 (원본·헬퍼·얼리스타트·보증인원). 나머지는 "더 보기" 뒤로.
3. **L1 활용 — 사용자가 이 카드를 닫으면 학습**
   - 사용자가 표준 체크리스트를 N번 무시(스크롤 패스)하면 자동 접힘 모드.

**우선순위**: ★ (낮음) — 위반 강도 작음. 다른 플로우 먼저.

- 응답: 마크다운 5컬럼 표 (업체·별점·후기·시작가·위치)

### v2 자기검문

| 항목 | 평가 |
|---|---|
| §1 수집 위계 | **L1 미활용** — 사용자가 명시 쿼리해야 발동. 행동 신호(찜·조회·체류)로 자동 트리거 가능. |

| Q7 직접 입력 안 받기 | ⚠️ — 현재는 키워드 입력 필요. 행동 누적 → 자동 카드 전환 가능. |

### 안티 패턴 진단
- **A7 Interrogation Mode 부분 위반**: 사용자가 직접 "비교해줘" 쳐야 발동. 같은 카테고리 N개 찜한 사용자에겐 자동 노출하는 게 v2 정신.

### 개선 제안

1. **L1 행동 신호 + L4 부드러운 확인 패턴**
   - 사용자가 같은 카테고리(예: 호텔 베뉴) **3개+ 찜**하거나 **5개+ 조회** 누적 시:
     - PersonaDashboard 또는 AI Planner welcome chip 에 카드 노출:
     - "찜한 강남 호텔 3곳을 비교해 보시겠어요?" `[비교 보기] [지금은 괜찮아요]`
   - 클릭 시 handleVenueCompare 와 동일 형식의 표 응답
2. **새 dbHandler 추가**: `venue_compare_favorites` — 인자 없이 user 의 찜 목록 기반 비교
3. **AI Planner welcome chip 동적화**
   - 현재 빠른 질문 카드가 일부 정적. 사용자 행동에 따라 비교 카드를 동적으로 한 슬롯에.

### 구현 변경 위치
- `src/lib/chatbot/handlers/searchHandlers.ts` — favorites 기반 비교 함수 추가
- `src/components/home/PersonaDashboard.tsx` 또는 `src/components/home/SuggestedActions.tsx` (있다면) — 비교 카드 노출
- `src/hooks/useFavorites.ts` 의 카운트를 활용

**우선순위**: ★★ (중간) — Q7 위반. 행동 신호 활용 자체가 v2 핵심.

## 3. F3 — 임산부 안전 허니문 (PregnancyHoneymoonBanner)

### 현재 상태
- `src/components/honeymoon/PregnancyHoneymoonBanner.tsx`
- 트리거: `weddingSettings.pregnant=true` + `pregnancy.trimesterAtWedding` 계산됨
- 노출: Honeymoon.tsx 페이지에 항상 (조건 충족 시)
- 컨텐츠: 차수별 비행시간 상한 + 추천 옵션 3개 + 주의사항 3개

### v2 자기검문

| 항목 | 평가 |
|---|---|
| §1 수집 위계 | 정보 수집 0 — 노출 콘텐츠. 단, `pregnant` 자체는 모달에서 받음 (L6). L1~L4 검토 미흡. |
| Q3 누구에게 | ✅ — 페르소나 자동 분기 (`pregnant=true`만) |
| §5 민감 정보 | ⚠️ — pregnant 동의는 모달에서 받지만 본 카드의 dismiss/forget 흐름 없음. §5.4 위반 |
| §5.5 카피 톤 | ✅ — "임신 초기 — 컨디션 보수적으로" 정보형. 객체화 아님 |

### 안티 패턴 진단
- **A8 Sensitive Cliff 부분 위반**: 한 번 pregnant=true 가 되면 Honeymoon 진입 시마다 본 배너 노출. dismiss 없음. 사용자가 "한 번 봤어요" 표시 못함.
- **A3 Form Essay 경계**: 추천 옵션 3 + 주의사항 3 = 약 200자. 가독성 한도 임계점.

### 개선 제안

1. **dismiss + 30일 cooldown** (§5.4 잊혀짐 응용)
   - X 버튼 추가 → localStorage `dewy:pregnancy-honeymoon-dismissed-at` 타임스탬프.
   - 30일 동안 같은 사용자에게 노출 안 함. 단 trimester 가 바뀌면 다시 노출 (정보가 달라지므로).
2. **요약 + 펼침 패턴**
   - 첫 노출: 헤드라인 + 비행시간 1줄 ("중기 안정기 — 6~8시간 비행권") + "자세히 보기" CTA
   - CTA 누르면 추천 옵션·주의사항 펼침. 200자 한도 회피.
3. **L1 자동 페르소나 보정**
   - 현재 `pregnant=true` 는 모달에서 받음(L6). v2 정신 따라:
     - 임산부 드레스 페이지 3회 조회 → §4.3 행동+확인 카드 자동 노출
     - "임신 중에 결혼 준비하시는 분들을 위한 일정·드레스·신혼여행 가이드 받기" `[받기] [지금은 괜찮아요]`
   - L4 패턴으로 받으면 본 배너 출현이 더 자연스러움.
4. **본식 지남 케이스 제거**
   - 현재 `trimesterAtWedding` 만 있으면 무조건 표시. wedding_date 가 지났으면(D+1 이상) 신혼여행 가이드 의미 적음.
   - 조건 추가: `daysUntilWedding > -30 && daysUntilWedding > -180` (산후 6개월 이내까지만).

**우선순위**: ★★ (중간) — §5 민감 정보 dismiss 누락이 가장 큼.

| Q7 직접 입력 안 받기 | ❌ — 디바이스 캘린더 권한·기존 스케줄 활용 가능했는데 안 함 |
| §6 위치 정보 | 해당 없음. |

### 안티 패턴 진단
- **A7 Interrogation Mode 명백 위반**: 사용자에게 직접 일자를 받는 게 유일한 흐름. v2 §1 위계 검토 안 함.
- **A6 Edit-Time Invisibility 부분 위반**: accordion 적용 안 됨. 가족 일정이 없을 때도 항상 펼친 상태로 노출. 가족 부모 정보(`has_parents_*`)는 있는데 일자는 없는 사용자에게 빈 상자가 매번 보임.

### 개선 제안

1. **L1 디바이스 캘린더 통합 (Capacitor)** ★★★
   - 사용자가 Capacitor `@capacitor/calendar` (또는 community plugin) 권한 허용 시 향후 8주 자유 일자 자동 가져옴.
   - JIT 권한 카드: "신부측 캘린더를 보고 빈 일자를 자동으로 채워드릴까요?" `[허용] [수동 입력]`
   - 한 번 허용 시: 향후 8주의 빈 토/일 자동 시드 → 사용자는 제외만 클릭.
2. **L3 기존 스케줄 활용**
   - `user_schedule_items` 중 카테고리 = `family_meet`(or `meetup`) 인 항목들의 일자를 충돌일로 자동 마킹.
   - 빈 일자는 "양가 모두 가능 후보" 로 자동 제시.
3. **L5 JIT 노출 전환**
   - 현재 Schedule.tsx 진입 시 항상 노출 → 다음 트리거 중 하나로 변경:
     - 일정에 "식장 투어" / "상견례" 추가 클릭 시 → 그 자리에서 인라인 노출
     - "양가 분담 시뮬레이션" 사용 후 → "일정도 맞춰볼까요?" 카드
   - Schedule.tsx 메인은 비어 있을 때 카드 자체 안 보임 (또는 작은 안내 한 줄로 축소)
4. **A6 회피: accordion 적용**
   - 양가 가용일이 0개인 신규 사용자에겐 `<details>` 접힘. 한 줄 카피: "양가 일정 조율 도구 — 양가 가능한 일자를 받아 식장 투어 후보 일자 찾기"
   - 입력값 있으면 자동 펼침.
5. **빈 상태 카피 강화**
   - 현재 placeholder "아직 없어요 — 아래 버튼으로 추가" 는 OK. 단, 캘린더 통합 시: "캘린더에서 자동으로 가져올까요?" 1차 CTA 우선.

### 구현 변경 위치
- `src/components/schedule/FamilyAvailabilityOverlap.tsx` — JIT 트리거·캘린더 통합·accordion

     - 임산부 드레스 N회 조회 → "임신 중이신 분들을 위한 가이드 받기"

     - "양가 분담" 카드 N회 dismiss → "혼자 진행하시는 분들을 위한 가이드 받기"
2. **국가 자동 추론**
   - 디바이스 locale 이 `ko-KR` → `country='KR'` 디폴트 + 확인 안 받음
   - locale 이 영어권/타지역 → "해외에서 결혼 준비하세요?" 카드 (L4)
3. **역할 자동 추론**
   - 가입 후 N일 사용 패턴에서 신랑/신부 카테고리 비율로 추론. 명시 입력 안 받음.
   - 추론 신뢰도 < 70% 면 명시 입력 옵션 제공.
4. **시군구 → 위치 권한 JIT**
   - 모달에서 빼고 베뉴/스튜디오 페이지 첫 진입 시 §6 패턴.
5. **식 형태 → L1**
   - 모달에서 빼고 wedding_style 선택만 받음. 사용자가 호텔·하우스·레스토랑 카테고리 자주 보면 ceremony_type 후보로 자동 채움.

- 필수: 결혼 예정일 / 예식 지역 / 진행 단계 / 결혼 스타일 (4)
